# 決済システム設計

> このノートでは決済システムの設計を扱う。重点は決済の接続、冪等性、二重実行防止、account記録、reconciliationと補償だ。本物のライセンスを持つ清算ネットワークを設計するわけではなく、決済リクエストと資金状態を中心に据えた業務システムを設計する話だ。

## この種のシステムが解決すべきこと
いわゆる「決済システムを設計する」というのは、たいていの場合 Visa や銀聯、あるいは銀行のコア清算ネットワークをゼロから作ることではない。もっとよくある範囲はこうだ。
- 加盟店がどう決済を起こすか
- ユーザーがどう支払いを完了するか
- システムがどう決済状態を記録するか
- どう二重課金を防ぐか
- どう記帳し、reconciliationし、補償するか

つまりこの種のシステムは、本質的に「**決済オーケストレーション + 状態管理 + account の正しさ**」という問題だ。

## 決済エコシステムの登場人物
典型的な決済フローには、だいたい次のような役者が出てくる。
- **カード保有者 / ユーザー**：決済を起こす人
- **加盟店**：受け取る側
- **発行銀行（issuer）**：ユーザーのカードの裏にある銀行
- **アクワイアラ（acquirer）**：加盟店側の収納銀行
- **カードネットワーク（Card Network）**：たとえば Visa、Mastercard
- **PSP（Payment Service Provider / Platform）**：統一された決済接続能力を外部に提供するプラットフォーム

なぜ多くの業務が PSP を使うのか。それは、加盟店が自前で異なる銀行やカードネットワークに直接つなごうとすると、こうなるからだ。
- 接続の作業量が大きい
- コンプライアンスの負荷が高い
- PCI 監査のコストが高い
- 各銀行の技術スタックやインターフェースのスタイルが統一されていない

だから多くのインターネット製品は、まず **PSP による統一接続**を選び、よりシンプルな API で決済リクエストを済ませる。代償としてサービス料がかかり、よくある水準はおおよそ **0.05% から 1%** くらいだ。

## 機能要件
基本的な決済システムなら、少なくとも次をサポートする必要がある。
- 加盟店が決済リクエストを起こす
- ユーザーが支払いを完了する
- 加盟店が決済状態を照会する
- 決済結果をユーザーや業務システムに通知する
- 後続の清算と reconciliation

## 非機能要件
決済システムでいちばん重要なのは、たいていスループットではなく、次のものだ。
- **安全性**：決済情報とユーザーデータを守る
- **正しさ**：accountと金が必ず一致する
- **一貫性**：同じ決済が互いに矛盾する状態を持ってはいけない
- **回復可能性**：外部システムの失敗、コールバックの欠落、ネットワークの揺らぎがあっても補償できる
- **監査可能性**：事後に追跡・照合・修正ができる

ここで一つ強調しておきたい。**決済システムはまず correctness first であって、latency first ではない。**

## ハイレベルなフロー
簡略化した決済フローは、こうまとめられる。
1. 加盟店が決済リクエストを作成する
2. 決済サービスが内部の payment レコードを生成する
3. PSP API を呼んで決済を起こす
4. PSP が受理結果、あるいは非同期処理中の状態を返す
5. システムが決済状態の変化を記録する
6. 最終的に webhook / callback、あるいは能動的な polling で結果を得る
7. 成功したら ledger に書き込む
8. 毎日あるいは定期的に reconciliation（対account）を行う

このフローでいちばん核心的な難所は「どう API を一本呼ぶか」ではなく、**どう非同期な状態変化、重複リクエスト、外部の不確実性を扱うか**だ。

## よくある決済の接続方式
### 1. 加盟店サーバーが銀行 / カードネットワークに直接つなぐ
これはいちばんコントロールが効く方式だ。加盟店はユーザー側から PAN / CVV などの機微情報を集め、バックエンドが直接アクワイアラやカードネットワークにつなぐ。

利点：
- 自主権が高い
- カスタマイズ度が高い
- サードパーティプラットフォームの一部の能力制限を回避できる
- 理屈の上ではサードパーティ経路の手数料を減らせる

欠点：
- 決済ライセンスや高いハードルの資格が必要
- PCI コンプライアンスと監査のコストがとても高い
- 実装が複雑で、安全責任のほとんどを自分で負う

この種の方式は、たいてい大企業しか検討しない。

### 2. PSP の SDK / JS Library を使う
これがインターネットでいちばんよくある接続方式だ。フロントエンドに PSP の JS SDK を組み込み、ユーザーがカード情報を入力すると機微情報は直接 PSP に渡り、加盟店は token を受け取る。そしてバックエンドがこの token で PSP API を呼んで課金を起こす。

利点：
- 接続が速い
- 加盟店が機微なカード番号を直接扱わない
- PCI 負担が明らかに低い

欠点：
- フロントエンドの SDK への依存が強くなる
- フロントエンドのサプライチェーン攻撃のリスクがある
- ページに悪意あるスクリプトが注入されると、決済データを盗まれる恐れが残る

そのためフロントエンド側では次を考える必要がある。
- **CSP（Content Security Policy）**
- **SRI（Subresource Integrity）**
- スクリプトのハッシュ検証
- iframe / sandbox による機微な入力欄の隔離

### 3. ホスト型決済ページへのリダイレクト
つまりユーザーが支払いをクリックしたあと、302 やページ遷移で PayPal、Stripe Hosted Checkout、WeChat Pay、Alipay などのホスト型ページに入る方式だ。

利点：
- 実装がいちばん簡単
- PCI の負荷がいちばん低い
- 運用と安全の負担が小さい

欠点：
- カスタマイズ度が低い
- ユーザーがサイトから離脱し、コンバージョン率が下がる可能性がある

### 一つの大事な結論
一言でまとめられる。**カスタマイズ度が高いほど技術とコンプライアンスのコストが高くなり、楽な接続方式ほど業務側のコントロール力が弱くなる。**

## データモデリング：大きな一枚テーブルだけで済ませない
決済の場面でいちばん犯しやすい設計ミスは、注文状態・決済状態・資金状態をすべて一枚の可変な大テーブルに詰め込むことだ。こうすると後から監査が難しくなり、状態がなぜ変わったのかも説明しづらくなる。

もっと筋のいいやり方は、三つのドメインを明確に分けることだ。
1. **業務 / 注文の状態**
2. **取引 / 決済の状態**
3. **資金 / account の状態**

だからより合理的なデータモデルは、たいてい少なくとも次を含む。
- **order table**：業務上の注文を記録する
- **payment table**：決済の試行と状態を記録する
- **ledger**：account の流れを記録する

### payment table のよくあるフィールド
決済レコードには、たいてい少なくとも次がある。
- `checkout_id` / リクエスト ID
- `order_id`
- `payment_method`
- `psp_id` あるいは外部の取引番号
- `status`
- `amount`
- `currency`
- `created_at`
- `updated_at`
- `idempotency_key`

ここでの肝はこうだ。**payment table は決済プロセスを記述する役割で、ledger は資金の事実を記述する役割だ。**

## 一貫性と二重課金の防止
決済システムで最大級の事故の一つが、**double charge（二重課金）**だ。

だから必ず **idempotency key** を持たせる。

### 冪等性の設計
よくあるやり方はこうだ。
- ユーザーが checkout ページを開いた時点で idempotency key を生成する
- クライアントがそれを決済フロー全体に貫かせる
- サーバーは重複リクエストを受けたら、この key で重複排除する
- 外部の PSP を呼ぶときも、できるだけ idempotency key を渡し、PSP 側にも重複排除させる

こうすればユーザーが、
- 決済ボタンを連打しても
- ページのタイムアウト後にリトライしても
- ネットワークの揺らぎでリクエストが重複して到達しても

システムは二重課金を避けられる。

## 決済の状態機械
簡略化した状態機械は、たいていこう書ける。
- `create`
- `start`
- `processing`
- `success`
- `fail`

説明はこんなふうにできる。
- `create`：payment レコードが作られたばかり
- `start`：システムが正当なリクエストを受け、PSP に決済を起こす準備ができた
- `processing`：PSP が受理したが、まだ最終結果を返していない
- `success / fail`：終端状態

### なぜ状態機械が重要か
それは PSP がたいてい**非同期**だからだ。つまり、
- リクエストを起こしてもすぐに最終結果が得られるとは限らない
- callback が遅れることがある
- callback が欠落することがある
- ローカルの polling と遠端の callback で前後の順序が一致しないことがある

だから状態遷移にはルールが要り、勝手に上書きしてはいけない。

## 非同期コールバック、polling、リトライ
本物の決済システムでは callback / webhook が当たり前だが、callback だけを信じてはいけない。

たいてい同時に次を用意する。
- **Webhook / Callback**：外部の PSP が能動的に結果を通知する
- **Polling / Query API**：callback が遅れたり欠落したときに能動的に照会する

### リトライ設計の四点セット
外部の PSP が時間内に最終結果を返さない場合、よくあるリトライ戦略には次を含める。
1. **timeout**：このリクエストをどれだけでタイムアウトと判定するか
2. **backoff**：リトライ間隔を徐々に伸ばす。典型は exponential backoff
3. **jitter**：ランダムな揺らぎを入れ、一斉リトライを避ける
4. **dead letter**：最大リトライ回数を超えたら、人手か非同期の補償キューに入れる

たとえば、
- 1 回目：1 秒
- 2 回目：2 秒
- 3 回目：4 秒
- 4 回目：8 秒
- さらに `±15%` の jitter を重ねる

## Late Callback 問題
古典的な問題がある。
- ローカルが長時間結果を得られず、先に決済を `fail` にする
- ところがしばらくして PSP が `success` の callback を押してくる

このとき単純に「最後に書き込んだ方を信じる」とやると、状態が乱れる。

よくある対処はこうだ。
- **単調な状態遷移**のルールを設計する
- 状態更新に**バージョン番号チェック**を入れる
- 終端状態の更新により厳しい検証を加える
- 最後に **reconciliation** で兜（保険）の修正をする

つまり決済システムは、その時々のオンライン状態だけで正否を判断してはいけない。**最終的な真実は、多くの場合 reconciliation から来る。**

## Ledger と記帳
決済が成功したあと、たいていは結果を account システムにも書き込む。

account システムは payment table とは思想が違い、ledger は次をより重視する。
- **double-entry bookkeeping（複式簿記）**
- **append-only（追記）**
- **不変**
- **監査可能**

なぜか。資金系のデータは勝手に上書きされるべきではないからだ。誤りを見つけたとき、正しいやり方はたいてい「古いレコードを書き換える」ことではなく、**訂正の仕訳**を一行追記することだ。

チームによっては、SQL インターフェース・監査能力・暗号化能力を備えた ledger 型データベースのような、より専門的な account / 監査向けストレージを使うこともある。

## 対account（Reconciliation）
決済システムでは、オンライン状態はゴールではない。**reconciliation こそが最後のヒューズだ。**

よくあるやり方はこうだ。
- PSP や銀行が周期的に settlement file / 対accountファイルを下ろしてくる
- システムは日次で daily reconciliation を行う
- 内部の payment / ledger と外部の清算結果を突き合わせる
- 不一致を見つけたら、自動補償や人手レビューを起動する

reconciliation の価値は、
- callback の取りこぼしを見つける
- 状態の誤書き込みを見つける
- 資金が入金されていない、あるいは二重入金されているのを見つける
- ローカルと外部システムのビューの不一致を見つける

ことにある。

## 二つの全体アーキテクチャの方向性
### 案 1：従来型の決済状態テーブル + 独立した ledger
これはいちばん成熟していて、いちばんよくある案だ。

特徴：
- 決済状態は主に payment table に置く
- 資金の事実は ledger に置く
- 理解と実装が比較的容易
- 運用コストが低い

ほとんどのチームに向く。

### 案 2：Event Store / Immutable Ledger スタイル
より過激な設計は、イベントストリームや append-only log を source of truth に据え、projection / view で照会ビューを生成するものだ。

典型的なキーワードは、
- event store
- CDC
- CQRS
- append-only DB / stream

利点：
- 正しさと監査可能性がより強い
- リプレイや修正にとても向く

欠点：
- システムの複雑度がはるかに高い
- チームのハードルが高い
- エンジニアリングの成熟度への要求が高い

だからこの方向性は触れておく価値はあるが、最適解だと決めつけてはいけない。

## 加点になる発展ポイント：リスク管理 / 反不正
もう一段深く話したいなら、決済フローの手前に risk / fraud service を一つ足せる。

実装はシンプルなものから複雑なものまである。
- ルールエンジン
- ブラックリスト / ホワイトリスト
- 統計的特徴
- 機械学習
- グラフモデル / GNN

ここでのトレードオフはこうだ。
- ルールは説明しやすく、コンプライアンスと監査に向く
- ML / グラフモデルは複雑な不正の検知が強いが、コストが高く、レビューも難しい

## まとめ
決済システム設計の本筋は「どう課金するか」ではなく、**どうやって金と account を常に一致させ続けるか**だ。だからいちばん語る価値があるのは、たいてい次の点だ。
- PSP 連携で決済をつなぐ
- idempotency key で二重実行を防ぐ
- 状態機械で非同期の決済フローを扱う
- ledger で資金の事実を保存する
- reconciliation で最終的な修正をかける

一言でまとめると、**決済システムはまず正しさ・冪等性・監査可能性・補償を解決するものであって、単に速さを追うものではない。**
