# Design Uber：需給マッチングプラットフォーム設計ノート

> このノートは Uber のようなシステムを設計した経験をまとめたものだ。核心となる洞察は、**Uber は本質的に「需給マッチングプラットフォーム」（供給側 × 需要側をマッチングするプラットフォーム）である** ということ。この抽象から見ると、配車、フードデリバリー、株式取引、GPU スケジューリングはすべて同じ種類の「ガワ替え」システムになる。

## first principle に立ち返る：これは需給マッチングプラットフォームだ
この種のシステムは「実物」に引きずられて考えすぎないこと。「配車」そのものだけを見つめていると、多くの点が思いつかない。first principle に立ち返る。

> 我々が設計しているのは **需給マッチングプラットフォーム** だ。

需給マッチングプラットフォームの核となる pattern は、システム全体を **供給側 / 需要側という2方向** に分け、いくつかの段階として表現することにある。

1. 双方向のイベント受信と永続化（供給側のドライバー、需要側の乗客）
2. リアルタイムなマッチング（match）
3. ドライバーの状態機の処理
4. マッチング失敗時のエラー処理と副作用

この「多段階」のメインラインを明確に語るほうが、ある特定のコンポーネントの細部にこだわるより重要だ。

## マッチングのいくつかの段階
### 1. 乗客の注文発行
乗客が注文を出すとき、2種類の情報を伴う。

- いくつかの要望（4人乗り / 6人乗りなど、細かい条件は後出し）
- **最も重要なのは出発地と目的地の位置情報**——この2つの位置を頼りにドライバーを探す

### 2. ドライバーの位置情報の報告
位置情報を頼りにドライバーを探すということは、ドライバー側に **高頻度の driver geo location 報告メカニズム** が必要で、定期的に位置を上げるということだ。プラットフォームはこの高頻度の受信をさばかなければならない。

よくある手は `in-memory cache` に位置を持つことだ（更新が速く、5秒ごとにリフレッシュ）——考え方は問題ないが、できれば **warm standby**（ホットスタンバイ二重化）を補っておくとよい。主 memory cache が落ちたときにシームレスに standby へ切り替え、5秒以内にリフレッシュし直す。

### 3. 位置情報ベースの検索 = indexing
位置情報データの検索というこの pattern の核となる概念は、**データの indexing**——二次元エンコードによる高速検索だ。最も重要なのは、この index をどう作るか。

- `Geohash (GH)`
- `S2`
- `H3`

これらは基本的に等価だ。

### 4. 注文をドライバーへ push：3つの選択肢
注文をドライバーへ push した後、ドライバーには3通りの反応がある：**受諾 / decline / 無応答**。これは非常に典型的な状態機で、そのまま雛形として使える。

## マッチングの制約：ロックをかける
matching の核となる制約はこうだ。

- **1つの注文は1人のドライバーにしか受諾されない**
- **1人のドライバーは同時刻に1つの注文しか占有できない**

外から見れば、これは **制約をかける = ロックをかける** ということ。

- データベースレベルでロックをかけるか、
- 上位レイヤーで distributed lock を行う

実務ではデータベースレベルでロックをかけることが多い。

## Notification：典型的な通信 pattern
通常、深掘りの一点を notification に置く。これが非常に典型的な communication pattern だからだ。

- ドライバーの位置報告 / プラットフォームからの push には **long polling** か **WebSocket** が使える
- 難しいのは scalability：peak 時にリクエストを取りこぼさないようどう保証するか——ここに **Queue** を1つ入れ、さらに BFF を入れるかどうかを検討する

## エラー処理：業務マッチングの失敗
これは需給マッチングプラットフォームなら必ず踏む点だ。供給側と需要側の業務マッチングが失敗したとき、error handling には次の選択肢を残しておく。

1. **動的リトライ（retry）**：単発のマッチング失敗は、まず差し戻してリトライする。
2. **技術的補償**：すでに一方に commit してしまった後で最終的に注文が失敗したら、技術的補償を必ず行う。
3. **業務的デグレード**：近くのこの candidate が受けなかったら、注文をより広い範囲に投げて他のドライバーと match させる。

### 副作用の処理
マッチング失敗 / デグレードは副作用を生むので、能動的に処理する。

- ユーザーに「もう少し待つ必要があるかもしれない」と通知する
- ログ側で「今回のマッチングは失敗した」と更新する
- **SLA 問題**：ユーザーに「1分以内にマッチング」と約束していた場合、それを守れなくなり、やむを得ず検索範囲を広げる
- **discount 補償**：以前提示した課金をそのまま続けられないことがあるので、ユーザーに「待ち時間が伸びたので割引します」と伝えて使い続けてもらう

## 価格で優先度を調整する
ごく自然に出てくる疑問：ある注文がずっとマッチングしないとき、matching service の中でその優先度を上げられないか？

first principle に立ち返る：**需給マッチングプラットフォームはどう優先度を調整するのか？** 技術レベルからではなく、生活の常識から考える——需給関係で優先度を調整するというのは、本質的に **価格設定** だ。

ライブ / 退勤ラッシュが始まるとき、プラットフォームは両側から調整をかける。

- ドライバーには「ここにホットスポット（hotmap）があり、より多くの注文を取れる」と伝える
- 乗客には「今はピーク時で、料金は ×1.2」と伝え、さらに「2ブロック先まで歩いたほうが捕まえやすく、料金も安い」と促す

「需給関係を使って双方に影響を与える」と腹落ちすれば、価格調整は優先度と均衡をまとめて扱う手段になる。

## ガワ替え問題：需給マッチングプラットフォームのいくつかの類型
なぜ Design Uber ≈ Design Stock Trading なのか？ 株式も ask price / bid price のマッチングだからだ。需給マッチングプラットフォームは大きく次のように分類できる。

1. **リアルタイムマッチング（real-time matching）**：配車、フードデリバリー、同一都市配送。本質は価格調整で需給に影響を与えること。
2. **検索予約型**：レストランのレビュー予約、会議室予約。
3. **情報マッチング**：医師 ↔ 患者、採用 ↔ 求職（linking プラットフォーム）。双方の意向をマッチングする必要がある。
4. **取引型**：広告取引（ad trading）、株式取引（stock trading）——価格設定、決済、リスク管理、紛争処理。
5. **リソーススケジューリング**：GPU のスケジューリング、物流のスケジューリング、Amazon が注文と物流をどうマッチングするか。

HLW の視点から見れば、これらはすべて需給マッチングプラットフォームだ。「供給側 / 需要側 / リアルタイムマッチング / マッチング失敗時の差し戻し補償 / 副作用の処理 / 価格調整」というこのテンプレートをそのまま移植すればよい。

## まとめ
Design Uber のメインラインは「どう配車するか」ではなく、**それを需給マッチングプラットフォームのマッチング問題に還元すること** だ。残しておく価値のある点：

- 双方向イベントの受信・永続化 → リアルタイムマッチング → ドライバーの状態機（受諾 / decline / 無応答）
- 位置情報の報告 + `geohash / S2 / H3` の indexing でドライバーを探す
- マッチング制約はロック（DB lock / distributed lock）で1注文1ドライバーを保証する
- notification は long polling / WebSocket、peak は Queue で取りこぼしを防ぐ
- マッチング失敗：動的リトライ / 技術的補償 / 業務的デグレード + 副作用処理（通知 / SLA / discount）
- 価格で需給の優先度を調整する

一言でまとめると：**「需給マッチングプラットフォーム」という抽象に気づければ、1つの解法を配車・株式・広告・スケジューリングなど、ガワ替えシステム一群にまるごと当てはめられる。**
