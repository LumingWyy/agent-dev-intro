# チャットシステム設計

> このノートでは典型的なチャットシステムの設計を扱う。メッセージの送受信、オンライン状態、マルチデバイス同期、オフライン補完がポイント。そして実務ではなぜ最初から P2P に飛びつくのではなく、クライアント・サーバモデルから話を始めるべきかについても触れる。

## チャットシステムが典型的な総合システムである理由
チャットシステムは多くのバックエンドの基礎課題をほぼ一通り串刺しにしている。
- ロングコネクション
- メッセージ配信
- オンライン状態
- マルチデバイス同期
- オフラインメッセージ
- 順序と一貫性
- スケーラビリティ

つまりこれをきちんと設計するということは、本質的に「リアルタイムシステム」の重要課題を一つずつ整理していくことに等しい。

## 基本的なチャットシステムが満たすべきもの
通常、少なくとも以下はサポートする必要がある。
- ユーザーによるメッセージの送受信
- 1対1チャット / グループチャット
- オンライン状態
- 履歴メッセージの取得
- マルチデバイス同期
- オフラインから再接続したときの未読メッセージの補完取得

もう少し現実的に考えると、さらに次のようなものも入ってくる。
- 既読通知
- プッシュ通知
- リスク管理・コンテンツ審査
- 取り消しと削除

## どのアーキテクチャから入るか
重要な経験則がある。**まずはクライアント・サーバモデル（Client / Server）を土台にし、いきなり P2P へ飛びつかないこと。**

理由は単純で、現実のチャット製品は通常どれも次を必要とするからだ。
- マルチデバイス同期
- 履歴メッセージの永続化
- オフラインメッセージ
- コンテンツ審査
- 通知とリスク管理

これらの要件を考えると、サーバ側はほぼ欠かせない存在になる。

## 典型的なハイレベルアーキテクチャ
よく登場するモジュールは次のとおり。
- **アクセス層**：ロングコネクション / WebSocket を担当
- **セッションサービス**：会話、グループ情報、メンバー関係を管理
- **メッセージサービス**：送信、配信、保存を処理
- **オンライン状態サービス**：ユーザーの現在の接続状態を記録
- **ストレージ層**：メッセージ履歴を保存
- **プッシュサービス**：オフラインユーザーへシステムプッシュを送る

## メッセージ送信の典型的な処理経路
1対1チャットを例にとると、送信フローはおおよそ次のようになる。
1. 送信側クライアントがロングコネクション経由でメッセージをサーバへ送る
2. サーバがメッセージ ID を生成して永続化する
3. サーバが受信側がオンラインかどうかを判定する
4. オンラインならリアルタイム配信に回す
5. オフラインなら履歴に保存し、必要に応じてプッシュ通知をトリガーする
6. ユーザーがオンラインになったら未読メッセージを補完取得する

ここで最も肝心になりがちなのは、**先にDBへ落とすか先に配信するか、順序をどう保証するか、オフラインをどう扱うか**だ。

## オンラインとオフラインの扱い方
### オンラインユーザー
- ロングコネクション経由で直接配信する
- レスポンスが速い
- リアルタイムなやり取りに向く

### オフラインユーザー
- メッセージは必ず永続化する
- モバイル端末へのプッシュ通知が必要になることもある
- ユーザーが再接続したときに補完取得できる必要がある

これこそが、チャットシステムを「リアルタイム送信」だけで語れない理由であり、メッセージの保存と同期まで踏み込まなければならない理由だ。

## マルチデバイス同期がなぜ重要か
現代のチャット製品では、一人のユーザーが同時に次のようなデバイスにログインしていることが多い。
- スマートフォン
- タブレット
- Web
- デスクトップ

そのためシステムは「メッセージが送れる」ことだけでなく、次も解決する必要がある。
- 同一アカウントの複数デバイスをどう同期するか
- 既読状態をどう同期するか
- メッセージ順序をどう統一するか
- あるデバイスがオフラインだったときに履歴をどう補うか

これも、サーバ中心の集中型モデルの方が現実的になる理由の一つだ。

## 順序と配信保証
チャットシステムで避けて通れない問題がいくつかある。
- at least once を保証するのか、それとも exactly once を目指すのか？
- 会話内では厳密な順序を要求するのか？
- どう重複排除するか？

現実にはより一般的なのは次のパターンだ。
- **at least once の配信**
- サーバがメッセージ ID と組み合わせて重複排除する
- 会話単位で可能な限り順序を保つ

なぜなら「真の exactly once」は分散システムではコストが非常に高いからだ。

## グループチャットが1対1より複雑になるところ
グループチャットは追加の問題を持ち込む。
- ファンアウトがより大きくなる
- メンバーリストがより複雑になる
- 一部のグループはホットスポットになる
- 同期と未読カウントがより難しくなる

そのため、いったんグループチャットに踏み込むと、自然に次のテーマへ広がっていく。
- ライトファンアウト vs リードファンアウト
- 大規模グループのホットスポット
- オンラインメンバーへのブロードキャスト
- メッセージストレージの sharding

## チャットシステムを P2P だけで語れない理由
P2P は特定のシナリオには向くが、主流の製品にとっては次のような弱点がよく見られる。
- マルチデバイス同期が難しい
- オフラインメッセージが作りにくい
- コンテンツ審査が難しい
- 企業・プラットフォームレベルの機能を実現しにくい

そのため、システムに明確な P2P 要求がないのであれば、サーバ中心の集中型を優先する方が通常は安定する。

## 設計をどう組み立てるか
自然な進め方の順序は、通常次のようになる。
1. まず機能要件を整理する
2. C/S のハイレベルアーキテクチャを説明する
3. メッセージの送信・配信経路を説明する
4. オンライン / オフラインを説明する
5. マルチデバイス同期を説明する
6. 順序、一貫性、重複排除を説明する
7. 最後にグループチャットの拡張問題を補足する

## まとめ
チャットシステム設計の核心は「どうやって一通のメッセージを送り届けるか」ではなく、ロングコネクション、メッセージの永続化、オンライン状態、マルチデバイス同期、オフライン補完を軸に安定したリアルタイム通信システムをどう築くかにある。実務ではまずクライアント・サーバモデルをしっかり据える方が、いきなり P2P を持ち出すより安定する。
