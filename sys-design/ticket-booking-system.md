# チケット予約システム設計ノート

> このノートは、チケット販売 / チケット争奪システムの設計経験をまとめたものだ。重点はコンポーネントを網羅して描くことではなく、**本当に成否を分ける数点をどう選び出すか、なぜボトルネックがデータベースにあるのか、そしてユーザー体験からどう逆算して方針を立てるか**にある。

## この種のシステムの本質的な難しさ
チケットシステムは見た目こそ「予約 / 争奪」だが、実際に難しいところはかなり限られている。

- `database selection` や `shard key` といった細部に労力を割きすぎて、システムの主軸を取りこぼしやすい
- 体験を本当に左右するのは、**強整合な注文確定、並行争奪の制御、非同期決済、ピークトラフィック** という数本のメインラインだ

だからこの種のシステムに取り組むときは、まず最初にやるべきは **優先順位をはっきりさせること**。どの数点が最大の成果につながるかを見極め、まずそこを固める。力を均等にばらまかない。

## 本当に固める価値のある数点
1. **なぜ relational database を使うのか**：争奪シーンは強整合が必要で、関係型データベースを選ぶ理由をきちんと説明できないといけない。
2. **distributed lock**：争奪の並行制御の核心。
3. **3rd party payment latency**：注文確定には決済が伴うが、決済は非同期な外部呼び出しだ。非同期決済を回しつつどう low latency を保つか、ここに本物の trade-off がある。
4. **不均一なトラフィック**：販売開始の瞬間、終了間際、ラスト5分の争奪——トラフィックはピーク型で、湧き出てきたリクエストを消化する何らかの仕組みが要る。

一方、API design のたぐいは、システムの高レベル構造に対して増分の情報をほとんど持たないうえに、やたら尺を食う——APIを並べているだけで軽く10分は過ぎ、本当にやるべきことを押しのけてしまう。よほど必要でない限り、展開しなくてよい。

## 高レベル設計をどう組み立てるか
### まず high-level walkthrough（HLW）を一通り通す
毎回 HLW を一度描き、主要な path を一度通すことを勧める。

- 自分で描いた箱でも、一度通してみないと何を漏らしているか案外わからない
- できれば high-level diagram を一枚描くとよい

このシステムには少なくとも2本の path がある。

- 読みパス（view part）：search → view event → 実際のダウンロード
- 書きパス：booking（注文確定）

### 段階的な確認と並べ替え
設計を進めるときは、1つの段階を話し終えるたびにいったん止めて **sanity check** を入れ、方向がずれていないか確認する。同時に、これからやる topic をいくつか並べ、システム内の重要度に従って順序を付け、high priority のものから先に片付ける。これはプロジェクトで ticket を並べるのと同じだ——リソースが限られているなら、まず重要なことを prioritize する。

## 不均一トラフィックの扱い：Queue を入れるべきか
よくある手は `API Gateway` と `booking service` の間に Queue を挟むことだ。だが入れる前に、まず **なぜ Q を入れるのか、それが何を解決するのか** をはっきりさせておく。

根本の問題は **トラフィックが不均一なこと**。販売開始と終了間際に大量のリクエストが押し寄せる。不均一なトラフィックである以上、湧き出てきたリクエストを消化する何らかの仕組みが少なくとも必要になる。

- **Q を入れる**：ピークトラフィックを buffer する
- **distributed lock / cache を入れる**：無効なリクエスト（90%は取れない）を前段で early rejection し、後段の負荷を軽くする

### Queue を入れる trade-off
実装に落とすときは損得をきちんと計算する。

- **利点**：ピーク時に湧き出るリクエストを buffer できる。ユーザーの待ち時間が増える代わりに、リクエストが捨てられないようにできる。
- **欠点**：いくらか latency が増える。コンポーネントが1つ増える＝システムの複雑度が1つ増える。
- **役に立たないケースはあるか**：ほぼない。システムは1つの event だけを相手にするわけではない——この event のトラフィックが落ち着くころには、別の event のピークが立ち上がってくる。だからここに Q を1つ置いておくのは大して問題にならない。

具体的に Q を使うか、redis か、lock かは **back-of-envelope estimation** で決める。ピークが `1M QPS` のオーダーなら、おそらく lock なしでは無理だし、Q なしでも無理だ。

## 本当のボトルネックはデータベースにある
event の開始 / 終了時に **booking service instance を preallocate して増やす**、あるいは schedule に沿って autoscale するという発想がある。これは緩和にはなるが、まずボトルネックがどこにあるかを見極める必要がある。

- フロント：リクエストを受け止めて捨てないだけの service があるか
- バック：データベースが受け止めきれるか

**本当のボトルネックは多くの場合、バックエンドのサーバーではなく DB にある。** relational database で consistency と availability を同時に保証しようとすると、それ自体が矛盾しているからだ（CAP）。バックエンドにいくら機材を足しても、リクエストは最終的に DB に到達する。そして DB が提供できる QPS には限りがあり——チケット1枚ごとに照会が走るため、リクエストを fast prove できない。だから booking service を preallocate するだけでは根本問題は解決しない。

## ユーザー体験から逆算する
最も肝心な一手は、**技術の細部から抜け出して、ユーザー体験から逆算すること**だ。

ラスト5分の争奪では、スレッドのレベルではもはや low latency を保証できない。このとき、ユーザーは何を望み、何を感じ取るべきかを考える。

- フロントはユーザーに **waiting list** を出す——ユーザーはそもそも、争奪では即座に結果が得られないと織り込んでいる
- waiting list に入ったユーザーは、しばらくして一通のメールを受け取る：booking 成功 / 失敗
- その後は **admission control** / 状態機で放行のしかたを決める——この部分はすでにユーザーに対して非同期になっている

技術の細部からユーザー体験へ逆算するのは、高並行シーンを設計するうえで非常に価値のある視点だ。まず極端なケースでの物理的限界を認め、そのうえでプロダクトの形（待ち行列 + 非同期通知）でそれを包む。

## まとめ
チケット予約システムで一番覚えておく価値があるのは、**どう主軸をつかみ、ボトルネックを見抜き、プロダクトの形で極端なケースを受け止めるか**だ。

- まず最大の成果点を prioritize する：relational DB の選定、distributed lock、3rd party payment latency、不均一トラフィック
- 不均一トラフィックは Queue で buffer するか distributed lock で early rejection し、trade-off をきちんと計算する
- 本当のボトルネックは DB（一致性 vs 可用性）であって、バックエンドサーバーを積み増すことではない
- 取れないときはユーザー体験から逆算する：waiting list + 非同期通知

一言でまとめると：**この種のシステムの勘どころは、正しい点を、正しい順序で、正しい深さまでやり切ること。そして極端な並行下ではプロダクトの形で受け止めるしかないと認めることだ。**
