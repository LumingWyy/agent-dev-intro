# ChatGPT システム設計

このノートでは ChatGPT / LLM サービスのシステム設計について、特に **推論サービスのスループット、レイテンシ、KV Cache、batching**、そして **prefill / decode を中心とした推論アーキテクチャの組み立て方** に焦点を当てて解説する。

---

## 難点

- 高い同時実行数の推論をどう支えるか
- 最初のトークンが返るまでの時間（**TTFT: Time To First Token**）をどう抑えるか
- スループット / レイテンシ / コスト のトレードオフをどう取るか
- コンテキスト / VRAM / KV Cache をどう管理するか

つまり本質的には **LLM inference serving system design** の問題である。

---

## ハイレベル・アーキテクチャ

よくある高レベル構成は以下のように分解できる:

- **API Gateway**: リクエスト受付、認証、レート制限
- **Session Manager**: セッションとコンテキストの管理
- **Conversation Storage**: 対話履歴の非同期保存
- **Message Queue / Task Queue**: 推論タスクのバッファリング
- **Inference Service**: 実際にモデル推論を行うサービス

リクエストはおおまかに以下のように流れる:

1. ユーザーリクエストが API Gateway に到達
2. レートリミットをかける
3. Session Manager が会話 ID からコンテキストを取得
4. リクエストを task としてキューに投入
5. Inference Service がタスクを消費し推論を実行
6. 結果をユーザーへ返却
7. 対話履歴は非同期で永続化

この構成のポイントは、**オンラインのリクエストパスと状態の永続化を分離する** ことで、回答生成のたびに DB 書き込みがブロッキング要因にならないようにすることである。

---

## 推論プロセスは内部で何をしているのか

モデル視点では、1 回の推論はおおむね以下のステップを経る:

1. **Tokenize**: prompt をトークン列に分割
2. **Embedding**: ベクトル空間にマッピング
3. **Transformer Layers**: 複数層の attention + feed-forward を通過
4. **Sampling**: 確率分布から次のトークンをサンプル
5. **Detokenize**: トークン列をテキストに復元

最も計算が重いのは通常以下の 2 つである:

- **Multi-Head Attention**
- **Feed-Forward Network (FFN)**

attention の部分には特に重要な **Q / K / V** が含まれる。

---

## GPT 推論の 2 つの基本的特徴

### 1. トークン単位の生成

LLM は基本的に「一気に全文を出す」のではなく、**1 トークンずつ生成する**。

### 2. 自己回帰 (autoregressive)

新たに生成したトークンはコンテキスト末尾に追加され、次のトークン予測に再び使われる。

つまり、**毎ステップで全コンテキストを再計算すると非常に高コスト** になる。

---

## なぜ KV Cache が重要なのか

ここで登場するのが **KV Cache** の価値である。

attention では、過去のトークンから計算した **K と V をキャッシュしておくことができる**。新しいトークンを生成する時:

- 過去のトークンの K/V を再計算しない
- 新しいトークンに対応する Q/K/V のみを計算
- 既存キャッシュと連結して使う

これにより重複計算を大幅に削減できる。

> **直感的な結論**
> KV Cache がなければ長いコンテキストの推論はほぼ実用に耐えない。KV Cache があって初めて、decode 段階が現実的な性能で動く。

---

## 推論の 2 つのフェーズ

実装上、推論は通常以下の 2 段階に分けて考える。

### 1. Prefill

完全な prompt を入力として、全コンテキストを計算し KV Cache を構築する。

**特徴:**
- 計算量が大きい
- TTFT への影響が大きい
- prompt が長いほど遅くなる

### 2. Decode

既存キャッシュを土台に、後続トークンを 1 つずつ生成する。

**特徴:**
- ステップごとの計算量は小さい
- ただし毎ステップで既存 KV Cache へアクセス
- 生成長が長いほど合計時間が増える

> **面接で必ず覚えておく指標**
> **TTFT (Time To First Token)** は主に **prefill** で決まる。一方、後続トークンのレイテンシは主に **decode** に支配される。

---

## なぜ VRAM がボトルネックになりやすいのか

推論サービスを設計する際、ボトルネックは **計算能力ではなく VRAM** であることが多い。

VRAM には通常以下を同時に載せる必要がある:

- モデルの重み
- KV Cache
- 実行時バッファ
- batching に伴う追加占有

### 典型的な見積もりの考え方

例えば A100 80GB 1 枚で 7B モデルを動かす場合:

- 重みを FP16 で持つと **約 7B × 2 bytes = 14GB**
- 32 層、hidden size 4096、context 4096 と仮定
- KV Cache は 1 トークンあたり **約 512KB** 程度のオーダー
- 4096 トークン分のキャッシュは **約 2GB** 程度

ここから分かるのは:

- モデル重みだけで VRAM の大きな塊を占める
- アクティブな session が 1 つ増えるごとに KV Cache が追加で必要
- **実際の並列上限は KV Cache に大きく制約される**

### Grouped KV Heads / GQA の意味

**Grouped-Query Attention** や **KV head 数の削減** を行えば、KV Cache のコストを大きく下げられるため、並列度を上げられる。例えば 8B モデルで KV head を減らすことで、1 session あたりの KV 占有量を **約 1GB 程度** まで抑えられるケースもある。

重要な面接ポイント:
> 並列上限は GPU 枚数だけではなく、「**重み + KV Cache**」を合わせて何 session 載るかで決まる。

---

## Batching: スループット最適化の中核

推論サービスでまず取り組む最適化が **batching** である。

### Naive batching の問題

単純にリクエストをまとめてから一括で実行すると:

- バッチが埋まるまで待ち時間が発生
- バッチ内で最も遅いリクエストに引きずられる

スループットは上がるが、**レイテンシ制御が効きにくい**。

### Continuous Batching

より実用的なのが **continuous batching**:

- 実行中に空きスロットへ新規リクエストを動的に追加
- 全リクエストを「同時開始・同時終了」にしない
- スループットとレイテンシを両立できる

現代の LLM serving では事実上の標準的アプローチである。

---

## なぜ長い prompt がシステムを壊すのか

長 prompt の問題は **prefill が重すぎる** ことにある。

continuous batching の中に超長 prompt が 1 つ混ざると、その大きな prefill 計算が多くの decode タスクを足止めし、全体のレイテンシ揺らぎが拡大する。

### 重要な 2 つの最適化

#### 1. Chunked Prefill

超長 prompt の prefill を **小さなチャンクに分割して実行** することで、中断しやすく、他タスクと交錯させやすくする。

#### 2. Prefill / Decode Disaggregation

**prefill と decode を別ノード / 別 GPU に分離** する。

メリット:

- prefill 専用機は大きな prompt を捌くことに専念
- decode 専用機は低レイテンシのトークン生成に専念
- 混在させるよりリソースチューニングがしやすい

例えば以下のような構成も考えられる:

- **1 台**: prefill ノード
- **2 台**: decode ノード

---

## PD 分離がもたらす新たな問題: KV の転送

prefill / decode を分離すると、**KV Cache をどう転送するか** という新たな核心問題が生じる。

A 機で prefill、B 機で decode を行う場合、キャッシュを A から B に転送する必要があり、この転送自体が追加レイテンシとなる。

### よくある最適化方向

#### KV サイズの削減

- FP16 → FP8 量子化
- paged attention / アクティブブロックのみ転送
- prefix sharing / prefix cache
- 圧縮、剪定、スパース化

#### 転送速度の向上

- **InfiniBand**
- **RoCE v2 / RDMA over Converged Ethernet**

#### 転送戦略の改善

- バッチ送信で小パケット濫発を回避
- 転送と計算を **overlap**
- layer streaming

見落としがちな現実的ポイント:
> **小さなパケット通信は効率が悪い**。場合によっては通信量が小さすぎてネットワークやフレームワークの低効率領域に落ち込む。

---

## モデルが大きすぎる時: Parallelism

GPU 1 枚に載らないほど巨大なモデルを動かす場合、並列化が必須となる。

### 1. Tensor Parallelism

**同じ層のテンソルを複数 GPU に分割** する。

例えば 70B モデルを FP16 で持つと約 **140GB** となり、1 枚の GPU には到底載らない。4 枚に分けると、1 枚あたり約 35GB の重みを担当する。

- **メリット**: 巨大モデルが動かせる
- **デメリット**:
  - 各層で同期 / all-reduce が必要
  - 通信オーバーヘッドが小さくない
  - 短いリクエストでは特に割に合わない

### 2. Expert Parallelism (MoE)

**Mixture-of-Experts** モデル向け。一度に一部のエキスパートのみがアクティブになるため、理論上は計算を節約できる。

問題点:

- エキスパート負荷が不均衡になりやすい
- 人気エキスパートがボトルネックになる
- 動的ルーティングとエキスパートのレプリケーションが必要

### 3. Context Parallelism

**超長コンテキスト** 向け。例えば 128K 以上の context を扱う場合、コンテキスト処理を複数 GPU / 複数ノードに分割する必要がある。

通常は **ring attention**, **flash attention** などと組み合わせ、超長 prompt の prefill レイテンシ削減を目指す。

---

## 量子化とハードウェア差

低精度サポート能力はハードウェアによって異なる。

例えば:

- **H100**: FP8 サポートが優れている
- 新しいハードウェアほどアグレッシブな低精度に対応
- **A100**: 同種の会話文脈ではより高精度設定で動かすケースが多い

この種の問題では「GPU を増やす」だけでなく、以下を明確に語る必要がある:

- 精度をどう選ぶか
- 重みをどう配置するか
- KV Cache をどう圧縮するか
- 並列度をどう計算するか

---

## このタイプの問題をどう答えると説得力が出るか

面接では「推論フレームワークの機能リスト」を読み上げるような答えにせず、以下を軸に展開する:

1. リクエストパスの全体像はどうなっているか
2. **TTFT は何で決まるのか**
3. **並列度を制限するのは何か**
4. 長コンテキストはなぜ高コストなのか
5. batching がスループットとレイテンシの双方にどう影響するか
6. **なぜ PD 分離や並列化が必要になるのか**
7. コスト / レイテンシ / スループットの三者をどうバランスするか

---

## まとめ

ChatGPT のシステム設計の難所は「チャット」ではなく、**LLM 推論サービスをいかに動かし、安定させ、コスト効率を出すか** にある。残しておくべきメインラインは以下:

- **API Gateway / Session Manager / Queue / Inference Service** でリクエストパスを構成する
- KV Cache が decode の性能を決定する点を理解する
- **prefill と decode を区別する**
- **continuous batching** でスループットを高める
- 長コンテキストでは **chunked prefill** と **prefill / decode disaggregation** を検討する
- 巨大モデルでは **tensor / expert / context parallelism** を組み合わせる

一言で言えば:
> **この問題の核心は、LLM プロダクトの問題を「VRAM・キャッシュ・スケジューリング・通信」の問題に還元することである。**
