## 一、コアとなる課題と背景

ライブ配信ルームのリアルタイム弾幕システムの設計と課題。

### シナリオの特殊性
- **トラフィックの爆発性**（Traffic Burst）：配信開始時や盛り上がりどころで瞬間的に急増し、落ち着く → 次の盛り上がり → 再び急増、という波を描く
- **データの偏り**（Skew）：少数の超大型ルームが主要なトラフィックを担うため、ピークを基準に設計しなければならない

### 規模感（ピーク基準で見積もり、難度は 100× に跳ね上がる）
- **1 億** 同時オンラインのライブ配信ルーム
- **2 億** 同時接続ユーザー

### コアとなる課題
- 極めて高頻度の書き込み操作
- 百万単位の視聴者が同時オンラインになることで生じる**大規模なファンアウト（Fan-out）**の圧力

---

## 二、Functional Requirements（機能要件）

- **コメント送信**：ユーザーがライブ配信ルームでリアルタイムに弾幕を投稿する
- **コメントのブロードキャスト**：一件のコメントをルーム内の全視聴者へリアルタイムに配信する（Fan-out）
- **ルームへの参加 / 退出**：ユーザーのオンライン状態（Session）を維持する
- **履歴の遡り表示**：新しく参加したユーザーが直近のコメント履歴（例：直近 50 件）を見られる
- **ギフト / 特効**：処理ロジックは弾幕と似ているが、優先度はより高い

---

## 三、Non-Functional Requirements（非機能要件）

| 指標 | 数値 / 目標 |
| --- | --- |
| エンドツーエンド遅延 | < 200 ms |
| 読み書き比 | 1:100 ~ 1:10,000（典型的なファンアウト比） |
| 可用性 | 単一ルームの障害が全サイトを巻き込んで落とさない |
| 一貫性 | 最終一貫性；メッセージの順序問題（Data Disorder）に対処する必要がある |
| メッセージ欠落の許容 | 極端なトラフィック下では「ベストエフォート」でよく、サンプリングによる縮退を許容する |

---

## 四、High-Level Architecture（高層アーキテクチャ）

```
[Client] ←→ [Gateway Server (WebSocket)] ←→ [Pub/Sub / Push Service]
                                                      ↑
[Comment Service] → [Storage (NoSQL)] ──────────────-┘
       ↑
[API Gateway]
       ↑
[Client]
```

| コンポーネント | 責務 |
| --- | --- |
| **Gateway Server**（アクセス層） | WebSocket のロングコネクションを維持し、メッセージを配信する |
| **Comment Service** | コメントを受信し、ディスクへ書き込み、Pub/Sub へ投入する |
| **Push Service / Pub-Sub** | パブリッシャーと膨大なサブスクライバーを疎結合にする |
| **Room Management** | 「ユーザー ↔ ライブ配信ルーム ↔ Gateway」のマッピングを維持する |
| **Storage** | コメントを永続化する（NoSQL） |

### データフロー
1. ユーザー A が弾幕を送信する
2. Comment Service が受信してディスクへ書き込む
3. Pub/Sub へ投入する
4. Push Service が当該ルームを購読している全 Gateway を見つけ出す
5. Gateway が WebSocket を通じてユーザー B へ配信する

## 五、Detailed Design（詳細設計）

### 1. ロングコネクションの技術選定

| 技術 | 長所 | 短所 |
| --- | --- | --- |
| **WebSocket** | 双方向リアルタイム、遅延が最も低い | コネクション維持のコストが高い |
| **Long Polling** | 互換性が高い | オーバーヘッドが大きく、遅延がやや高い |
| **SSE** | 単方向でシンプル | サーバ→クライアントのみ |

**結論**：弾幕シナリオでは **WebSocket** が第一候補。

### 2. ルームの分片（Room Sharding）

- **分片キー**：**Livestream ID** を使用
- **理由**：同一ルームのユーザーは論理的な親和性を持ち、一括ブロードキャストに都合がよい

### 3. 多層ファンアウトアーキテクチャ（Multi-layer Fan-out）

百万単位のオンラインルームでは、単一の Gateway で全コネクションを支えきれない。

```
            [Comment]
                ↓
        [L1 Distributor]
          ↓        ↓
   [L2 Group A]  [L2 Group B]   ...
     ↓    ↓        ↓    ↓
  [Gateway][Gateway][Gateway][Gateway]
     ↓                  ↓
   ...users          ...users
```

層を増やすことで、任意の規模の視聴者を支える。

### 4. ユーザー親和性（Affinity）

| 種別 | 意味 |
| --- | --- |
| **物理的親和性** | ユーザーの地理的位置に応じて最寄りの PoP（Point of Presence）に割り当てる |
| **論理的親和性** | 同一ルームのユーザーをできるだけ同一グループの Gateway に振り分け、購読 Channel 数を減らしてブロードキャストを最適化する |

### 5. メッセージストレージ

- 高並行の書き込みシナリオでは、**NoSQL（Cassandra / DynamoDB）**がリレーショナル DB より優れる
- 書き込みスループットが高く、ピークをよりうまくさばける

### 6. ホットルームの処理

- 分散的な発見機構によって超大型ルームを動的に識別する
- ホットルームには専用のブロードキャストリソース（Dedicated Nodes）を割り当てる

### 7. 履歴メッセージのロード（重要テクニック）

- ❌ 新規ユーザーごとにデータベースへ問い合わせない（コストが高い）
- ✅ Pub/Sub のブロードキャスト時に、新規入室ユーザーへ直近 50 件のスナップショットをそのまま添付する

## 六、Scaling & Bottlenecks（スケーリングとボトルネック）

| ボトルネック | 解決策 |
| --- | --- |
| **コネクション数のボトルネック** | 単一マシンの WebSocket 上限は約 200 万；Gateway ノードを水平スケールする |
| **ホット Key**（超大型ルーム） | sharding + 多段キャッシュ + 専用ブロードキャストリソース |
| **ネットワーク帯域**（千万単位のメッセージ/s） | メッセージ圧縮、サンプリングによる縮退（Sampling） |
| **書き込みの洪水** | メッセージキューへ非同期書き込みし、下流の DB がボトルネックになるのを避ける |

---

## 七、Trade-offs（トレードオフ）

| 観点 | 判断 |
| --- | --- |
| **正確な配信 vs ベストエフォート** | 超大型ルームでは、システムを崩壊させるくらいなら、良質な弾幕を 10% サンプリングして配信する方を選ぶ |
| **履歴ロード** | ブロードキャスト時にスナップショットを添付する > ユーザーごとに DB を個別に問い合わせる |
| **同期 vs 非同期** | 書き込み操作は必ず非同期化（メッセージキュー）し、下流の DB を保護する |
| **強一貫 vs 最終一貫** | 弾幕シナリオでは最終一貫性を受け入れるが、順序の乱れには対処する必要がある |

## 八、キーとなる用語

| 用語 | 意味 |
| --- | --- |
| **Fan-out（ファンアウト）** | 一件のメッセージを複数の受信者へ分配すること |
| **Consistency Hashing（一貫性ハッシュ）** | ノードの増減時におけるルーティングの安定性を扱う |
| **Affinity（親和性）** | 関連するリクエスト/コネクションを同一の処理ユニットに割り当てること |
| **Skew（データの偏り）** | 少数のルームが大部分のトラフィックを担う非均一な分布 |
| **PoP（Point of Presence）** | エッジのアクセスポイント |

## 九、実践の振り返り：よくある落とし穴と経験則

### よくある落とし穴
1. **アーキテクチャの階層を混同する**
   - High-level なアーキテクチャと Low-level な実装を一緒くたにしてしまう
   - ✅ 経験上：まず全体構造を整理する → それから細部に踏み込む
2. **方式の比較が欠けている**
   - WebSocket vs Long Polling を選ぶ際に Trade-off 分析をしない
   - ✅ これは設計ドキュメントで見落とされやすい一点だ
3. **既成の方式を盲目的に当てはめる**
   - データ規模がいったん変われば（Sanity Check）、推論を経ていない設計は成り立たなくなる
   - 設計が不自然で説明がつかないなら、たいてい本当に考え抜けていないことを示している
4. **データの偏りを無視する**
   - 平均トラフィックだけを考え、配信ルームの爆発性と Skew を考えない
   - ✅ 設計は必ずピークに向けて冗長を確保しておく

### 経験則
- まずトラフィックパターン（爆発 / 偏り）を明確にしてから設計する
- ロングコネクションの技術選定では WebSocket / SSE / Long Polling を明示的に比較する
- 多層ファンアウト + ルーム分片 + 親和性が、ライブ弾幕の「三種の神器」
- 履歴メッセージは DB ではなくスナップショットで運ぶ
- 極端なトラフィック下では 100% 配信を追わず、サンプリングによる縮退を受け入れる

## 十、システム設計への示唆

- **大規模ファンアウトシナリオのコア三点セット**：ルーム分片 + 多層 Fan-out + ユーザー親和性
- **ロングコネクションサービス設計のポイント**：コネクション数のハードな上限により、水平スケールは必須選択肢になる
- **ピークは平均に優先する**：ライブ系の業務はピーク/Skew を基準に設計しなければならず、平均 QPS は誤解を招く指標だ
- **優雅な縮退 > 全量の正確さ**：超大規模なリアルタイムシステムでは、サンプリング・破棄・レート制限は正当なツールだ
- **スナップショットで DB 負荷を減らす**：ブロードキャストプロトコルそのものが安価な「history fetch」の運搬役になり、クライアントごとに DB を個別問い合わせるより安い
- **設計の組み立て**：まず全体アーキテクチャを整理してから細部に踏み込む；どんな技術選定も Trade-off を明示的に議論する
