# CI システム設計

> このノートでは CI / Workflow 実行システムの設計を扱う。重点は control plane と execution plane の分離、タスクのオーケストレーション、タスクの配信、マルチテナントの拡張、そしてログと成果物の回収だ。

## この種のシステムが解決すべきこと
CI / Workflow システムは、単に「スクリプトを走らせる」だけのものではない。本当に解決すべきは次のことだ。
- コードリポジトリや外部システムのイベントを受け取る
- workflow run を生成する
- 依存関係を DAG に組み立てる
- job を runner に配信して実行させる
- 状態、ログ、成果物を回収する
- マルチテナントの場面で隔離と拡張性を保証する

だからこの種のシステムは、本質的に一式の **event-driven orchestration + distributed execution** システムだ。

## 機能目標と非機能目標
### 機能目標
基本的な CI / Workflow システムは、たいてい次をサポートする必要がある。
- 外部イベントを受け取る
- workflow run を作成する
- 複数の job をオーケストレーションする
- 依存と実行状態を追跡する
- ログと結果を回収する

### 非機能目標
よくある要求は次のとおりだ。
- **Scalability**：増え続けるタスク量を処理できる
- **Low Latency**：タスク状態をできるだけリアルタイムに近くフィードバックする
- **隔離性**：テナントごと、runner ごとに互いに影響しない
- **可観測性**：タスクの状態、ログ、失敗理由がすべて見える

この種のシステムは、たいてい従来型の完全な非同期ブラックボックスではない。ユーザーは「今どのステップまで進んだか」を見たいので、非同期実行と近リアルタイムの状態同期を両立させる必要がある。

## 二層のコアアーキテクチャ：Control Plane vs Data Plane
この種のシステムでいちばん肝心な構造は、システムを二層に分けることだ。

### 1. Control Plane
コントロールプレーンは主に次を担う。
- タスクを受け取る
- workflow を解析・オーケストレーションする
- 状態機械を管理する
- どの job をスケジュールできるかを決める
- タスクを runner に割り当てる

### 2. Data Plane / Execution Plane
実行プレーンは主に次を担う。
- 実行待ちのタスクを引き取る
- 実際に job を走らせる
- 実行状態を報告する
- ログと結果を回収して返す

### なぜ必ず分けるのか
オーケストレーションのロジックと実行のロジックを一緒くたにすると、
- スケールアウトが難しい
- runner の障害がコントロールロジックを巻き込みやすい
- マルチテナントの隔離がさらに難しくなる

だから経験上、一本の線は必ず守る。**コントロールプレーンと実行プレーンの分離が、このシステムの第一原則だ。**

## タスクはどこから来るか：Event Listener / Webhook / Event Bus
CI システムのタスク入口は、たいていコードリポジトリや外部イベントから来る。たとえば、
- push
- pull request
- tag
- release
- 手動トリガー

### 入口案 1：Event Listener / Webhook
いちばん直接的なやり方は、Git プラットフォームや上流システムが webhook で直接イベントを押してくることだ。

利点：
- シンプル
- 低レイテンシ
- 実装しやすい

### 入口案 2：Event Bus / Streaming / Change Feed
システムが後々より複雑なルーティング、監査、優先度制御を必要とするなら、入口の後ろに event bus を一つつなげる。たとえば Kafka だ。

単独の event bus 以外に、次も検討できる。
- **change feed**
- **Streaming**
- 一部 NoSQL のネイティブなストリーム能力、たとえば DynamoDB Streaming

### どう選ぶか
業務がシンプルで、フローが明快で、規模もまだ手に負える範囲なら、直接 NoSQL かメタデータテーブルに落とし、change feed を使って下流をトリガーする、でも十分かもしれない。

だが次のような要求がすでに明確なら、
- 複雑なルーティング
- 監査 / replay
- テナントごとの優先度
- より豊富な拡張能力

event bus のほうが合理的だ。

## マルチテナントがなぜ設計を変えるのか
この種のシステムでよくある追加の難所が **multi-tenant** だ。

テナントごとに、たいてい次が異なる。
- タスク量
- SLA
- runner の割り当て枠
- 優先度

たとえば大口顧客は次を必要とするかもしれない。
- より高い優先度
- より多い並行枠
- 専有の runner プール

このとき、メッセージシステムとスケジューラは単なる「先入れ先出し」ではなく、次をサポートする必要がある。
- tenant ごとの割り当て枠で限流する
- 優先度でスケジュールする
- runner のラベルでルーティングする
- 必要に応じてテナント隔離をする

だからマルチテナントは、event bus・優先度キュー・リソース隔離を支える重要な理由になる。

## Workflow の本質は DAG
CI システムでは、一つの workflow はたいてい単一タスクではなく、複数の job からなる DAG（有向非巡回グラフ）だ。

たとえば、
- まず build
- 次に test
- test が通って初めて deploy できる

だからスケジューラは「タスクがあるかどうか」だけでなく、次を見る必要がある。
- どの job の前提依存が満たされたか
- どの job がまだ待機しているか
- どの job が runnable 状態に入れるか

## メタデータモデリング
この種のシステムのデータモデリングは非常に重要で、大きな JSON を一つ保存するだけでは済まない。

### 1. workflow run
このテーブルは一度のワークフロー実行を記述する。よくあるフィールドは次のとおりだ。
- `run_id`
- `tenant_id`
- `repo`
- `commit_sha`
- `event_type`
- `status`
- 各種タイムスタンプ

### 2. job
各 workflow run は複数の job を含む。job テーブルにはたいてい次が要る。
- `job_id`
- `workflow_run_id`
- `name`
- `runner_id`
- `status`
- タイムスタンプ
- 依存に関わる必要な情報

### 3. dependency edge
workflow が DAG なら、依存関係は明示的にモデル化するのがよい。とても自然なやり方は、エッジテーブルを一つ持つことだ。
- `from_job_id`
- `to_job_id`

こうすると次が楽になる。
- ある job が実行可能かどうかを判定する
- 状態を前に進める
- 複雑な依存グラフをサポートする

## ストレージはどう選ぶか
### 案 1：Sharded Relational Database
たとえば PostgreSQL / MySQL のシャーディングだ。

利点：
- モデルが明快
- トランザクション能力が成熟している
- workflow / job / dependency のような強い関係を持つデータの記述に向く

欠点：
- シャーディング後、シャードをまたぐトランザクションと照会が複雑になる

### 案 2：NewSQL / Spanner の類
この種の案の強みは次だ。
- 分散トランザクションがより自然
- 横方向の拡張能力がより強い
- シャードをまたぐ一貫性の問題を自分で抱え込まずに済む

欠点：
- コストが高い
- 運用とチューニングのハードルが高い
- チームの成熟度への要求が高い

### 典型的なシャーディングのトレードオフ
シャード化した relational DB を使う場合、よくある問題はこうだ。**workflow id でシャードするか、それとも tenant id でシャードするか。**

- **workflow id でシャード**：同じ workflow のデータがより集中し、内部の一貫性がよくなる
- **tenant id でシャード**：テナントごとの履歴閲覧、割り当て枠、リソース隔離に向く

マルチテナントの CI システムでは、`tenant_id` でシャードするほうが実際の業務管理のやり方に近いことが多い。ただし一部の workflow をまたぐ照会の複雑度は増す。

## タスクの配信：Push vs Pull
コントロールプレーンがタスクを実行プレーンに渡すとき、よくあるモードが二つある。

### 1. Push モード
コントロールプレーンが能動的にタスクを runner に押す。

利点：
- 直感的
- レイテンシがより低い可能性がある

欠点：
- runner が NAT / ファイアウォールの裏にいることがある
- セルフホスト runner の環境が複雑
- システムの結合がより強くなる

### 2. Pull モード
runner が能動的にコントロールプレーンへタスクを取りに行く。

利点：
- セルフホスト runner により向く
- NAT / ファイアウォールを越えやすい
- コントロール面と実行面がより分離される

だから本物のシステムでは、**pull のほうが push より実用的なことが多い**。

### pull の実装ディテール
よくある手段は次のとおりだ。
- **long polling**
- `60s` 前後の lease
- タスクを取れなかったときは **exponential backoff** する
- `pull + notify` や WebSocket のバリエーションもできる

これらの仕組みの狙いは、runner にむやみに空回りの polling をさせず、かといってタスクの配信が遅すぎることもない、という両立だ。

## Runner の実行隔離
CI システムでは、runner はいちばん危険な層だ。なぜならユーザーのコードを直接実行するからだ。

よくある隔離方式は次のとおりだ。
- **コンテナ**
- **軽量 VM / MicroVM**

よくあるやり方はたとえば、
- Firecracker のような microVM の発想
- より強い安全隔離
- 起動速度とリソースオーバーヘッドのトレードオフ

システムが安全性やマルチテナントの強い隔離を重視するなら、runner の隔離は重点的に設計すべき層だ。

## ログ、状態、成果物の回収
成熟した CI システムは、job が成功か失敗かを知るだけでなく、ユーザーに次を見せる必要がある。
- 今どのステップまで進んだか
- リアルタイムのログ
- 最終的な完全なログ
- ビルド成果物

### よくあるやり方
- **小さな状態**：コントロール経路で素早く回収する
- **大きなログ / 成果物**：直接オブジェクトストレージに落とす
- presigned URL や類似の仕組みで成果物をアップロード / ダウンロードする

こうすれば、大量のログ内容をコントロールプレーンのデータベースに押し込むのを避けられる。

## 設計一式をどう組み立てるか
経験上、より成熟した設計の進め方は、たいてい次の順番だ。
1. まず機能要件と非機能要件を整理する
2. コントロールプレーンと実行プレーンを分ける
3. workflow が DAG であることを説明する
4. workflow run / job / dependency のメタデータモデリングをする
5. ストレージとシャーディングを決める
6. runner の pull、lease、backoff を設計する
7. 最後にログ、成果物、マルチテナントの拡張を扱う

## まとめ
CI / Workflow システム設計の核心は「スクリプトを走らせる」ことではなく、次のことだ。
- **Control Plane** でオーケストレーション・状態・スケジューリングを管理する
- **Data Plane** で実行と結果の回収を管理する
- **workflow run / job / dependency** で DAG をモデル化する
- **pull モード**で runner の接続を支える
- オブジェクトストレージでログと成果物を受ける
- マルチテナントの場面で優先度・割り当て枠・隔離を扱う

一言でまとめると、**これは DAG オーケストレーションを核心に、コントロール面 / 実行面の分離を骨格にした分散タスク実行システムだ。**
