## 一、核心となる課題と背景

ユーザーのテキストプロンプト（Text Prompt）に基づいて高品質な動画を生成するサービスを設計する。対標は OpenAI Sora。

### 従来の動画 / 画像システムとの本質的な違い
- **計算量のオーダーが跳ね上がる**: 動画生成の FLOPs は LLM をはるかに上回る
  - 比較: Llama 3 70B が一文を生成するのに約 **140 Tflops**
  - Sora の 16 秒 720p 動画 1 本: **~220 Petaflops** —— 6 桁高い
- **GPU は希少 / 高価なリソース**: あらゆるアーキテクチャ上の決定は「いかに GPU を節約するか」を軸に据える
- **時間次元が新たな難所を持ち込む**: 画像生成は空間だけを考えればよいが、動画はさらに **時間一貫性** と **物理的妥当性** を保証しなければならない

### 設計上の難所
| 難所 | 説明 |
| --- | --- |
| 時間一貫性 | 物体が再生中にランダムに変形したり消えたりしてはいけない |
| 物理的妥当性 | 重力の向き、鏡面反射、光と影のロジックなど |
| 鮮明度 | 最低 720p（モバイル端末）、エンタープライズ級 1080p |
| リソースコスト | GPU が極度に高価で希少 |
| 法的コンプライアンス | 公人の肖像、暴力、IP 侵害（ディズニー等）を拒否。AI 透かしを強制 |

## 二、Functional Requirements（機能要件）

- **核心機能**: ユーザーがテキスト prompt を入力 → システムが対応する動画を生成
- **動画スペック**:
  - 標準時間: 15 秒
  - エンタープライズユーザーは拡張可能: 60 秒、1080p

## 五、High-Level Architecture（高レベルアーキテクチャ）

```
[Client]
   ↓
[API Gateway]              ← Rate Limit, Quota Check, Auth
   ↓
[Backend Server]           ← 创建 Task, 维护状态机
   ↓
[Task DB]  ←──────────────  状态持久化
   ↓
[Priority Message Queue]   ← Free / Paid / Enterprise 三级优先
   ↓
┌──────────────────────────────────┐
│  Worker Pool                      │
│  ├─ Preprocessing (CPU)           │
│  ├─ Model Inference (8× H100 GPU) │
│  └─ Post-processing (small GPU)   │
└──────────────────────────────────┘
   ↓
[Object Storage (S3-like Blob)]
   ↓
[CDN (Adaptive Downloading)]
   ↓
[Notification Service] → [Client]
```

| コンポーネント | 役割 |
| --- | --- |
| API Gateway | リクエスト受付、限流、配額照会、サブスクリプション検証 |
| Backend Server | タスク生成、ステートマシン維持 |
| Priority Queue | ユーザー等級でソートしたタスクキュー |
| GPU Worker Pool | 3 段の pipeline、それぞれ異なるハードウェア |
| Object Storage | 大量の小ファイル、Blob/S3 |
| CDN | グローバル配信 + 自適応解像度 |
| Notification Service | タスク完了を非同期でユーザーへ通知 |

## 六、Workflow DAG（詳細ワークフロー）

生成フロー全体の三大ステージ + 時間予算:

```
[Prompt 输入]
     ↓
─────────────  Preprocessing (CPU / 小 GPU, ~几百 ms)  ──────
  ├─ 文本安全审核（关键词/IP 拦截）
  ├─ Prompt Enhancement（Text Rewriter 扩写）
  └─ 重写后再次安全审核
     ↓
─────────────  Model Inference (8× H100, ~30s)  ────────────
  ├─ Text Encoder：text → embedding
  ├─ Diffusion Transformer (DiT)
  │     30B 参数，50 个 denoising steps，在 latent space 去噪
  └─ Video Decoder (VAE / TAEF)：latent → pixel
     ↓
─────────────  Post-processing (单张旧 GPU, 较短)  ──────────
  ├─ 超分辨率 (SR)：720p → 1080p
  ├─ 插帧 (Frame Interpolation)
  ├─ 视觉层最终安全检查
  └─ 加水印（显性 / 隐形 / C2PA）
     ↓
[Object Storage] → [CDN] → [Push Notification]
```

## 七、DiT モデルの核心 — 系列長と FLOPs の導出

### 1. 系列長 73K はどこから来るのか?

```
原始视频: 16 s × 16 fps = 256 frames
分辨率:   768 × 768
                ↓ VAE 时空压缩 (8 × 8 × 8)
Latent:   96 × 96 × 32       (空间 96×96，时间 32)
                ↓ Patch Tokenization (2 × 2 × 1)
Tokens:   (96/2) × (96/2) × (32/1)
       =  48 × 48 × 32
       =  73,728 ≈ 73K
```

### 2. モデルパラメータ

| 項目 | 値 |
| --- | --- |
| パラメータ数 | **30B**（300 億） |
| Denoising Steps | **50** |
| 系列長 n | **73K** |
| 単回推論時間 | **~30 秒** |

### 3. FLOPs の計算

**厳密な公式**:
```
FLOPs ≈ 24 · n · d² · L · s
```
ここで n = 系列長、d = 隠れ次元、L = 層数、s = 去噪ステップ数。

**見積もり公式（パラメータ数法）**:
```
FLOPs ≈ 2 × 参数量 × 序列长度 × 步数
      = 2 × 30B × 73K × 50
      ≈ 220 Petaflops
```

### 4. Self-Attention の計算量
- 計算量: **O(n²)**
- n = 73K → n² ≈ **5.3 × 10⁹** のオーダー → 系列並列で最適化するしかない

## 八、GPU 選定と Pipeline のリソース割り当て

### 1. ハードウェア
| ステージ | ハードウェア | 显存 | 備考 |
| --- | --- | --- | --- |
| Preprocessing | CPU または小型 GPU | — | テキスト系の操作、数百 ms |
| **Model Inference** | **8× H100** | 80 GB / 卡 | 同時に占有し同時に解放 |
| Post-processing | 旧型 1 枚 / 小 GPU | — | コスト節約 |

### 2. 単回推論のリソース占有
- 1 台の 8-GPU H100 ノード
- 所要時間 ~30 秒
- 30B モデルの推論時、**中間活性化値 ~60 GB**（显存占有の大半を占める）

### 3. 系列並列 / USP（Unified Sequence Parallelism）

73K の長系列がもたらす显存と計算の圧力を最適化する。

- **核心思想**: Self-Attention、Cross-Attention、Feed-Forward の 3 層で統一的に並列化し、本来 3 回独立していた通信（Reduce 操作）を **1 回にまとめる**
- **本番運用の組み合わせ**: **CP4 + TP2**
  - CP4 = Context Parallelism × 4
  - TP2 = Tensor Parallelism × 2

### 4. 中間活性化 60 GB の影響

> これはシステム設計全体で最も反直感的で、最も重要な決定ポイントの一つだ。

- 60 GB の中間活性化値が意味するのは: それを保存する IO コスト > 30 秒の推論をやり直すコスト
- **結論**: **タスクの抢占をサポートしない / 断点続伝をサポートしない**
- いったん推論ステージに入ったら、その 30 秒を走り切らせるほうが、「中断 + 保存 + 復元」よりも割に合う

## 九、Admission Control（準入制御の三段階）

GPU が高すぎるため、何重もの関門を設けなければならない。

### Stage 1 — Rate Limit（限流）
- ユーザーの **Quota** と **Subscription Plan** をチェック
- 無料ユーザー: 限度が低く、優先度も低い
- 有料ユーザー: 高優先度
- エンタープライズユーザー: BYOM/BYOC（自前マシン/キャパシティ）を享受可能

### Stage 2 — Resource Check（リソースチェック）
- 空き GPU があるか確認
- リソース不足 → キューイング / 拒否

### Stage 3 — Traffic Shaping（トラフィックシェイピング）
- 優先度キュー: Free / Paid / Enterprise の 3 段階
- **リージョン調度**: 北米のピーク → 深夜帯の欧州データセンターへ調度し、時差を活用

### タスクステートマシン
```
Ready → Scheduling → Running → Complete
                        ↓
                       Fail → Retriable → Retry → Ready  (循环)
                        ↓
                     Final Fail (达到最大重试次数)
```

## 十、Storage 設計

### 1. 動画ストレージ — Blob / Object Storage
- **選定**: S3-like Object Storage
- **理由**: 大量に断片化した小ファイルのシナリオでは、Object Storage がファイルシステム（HDFS）より優れる
- CDN と直結し、グローバル配信

### 2. タスクメタデータ
- リレーショナル DB または NoSQL
- フィールド: TaskID、UserID、Prompt、Status、CreatedAt、CompletedAt、VideoURL など

### 3. 中間生成物
- DiT の中間活性化 60 GB —— **永続化しない**。走り切ったら即解放

## 十二、Safety Pipeline（前後 2 段の安全審査）

### 前置審査（Preprocessing ステージ）
- **遮断方式**: キーワードマッチ + テキスト分類
- **遮断対象**:
  - 公人 / 著名人の肖像
  - 暴力コンテンツ
  - 特定 IP の著作権（ディズニー等）

### 後置審査（Post-processing ステージ）
- **遮断方式**: **生成された動画コンテンツ** に対して視覚層の最終チェック
- **遮断対象**: モデルが生成時に持ち込んだ不適合な細部

### Prompt Enhancement
- **Text Rewriter** で扩写: 「猫が走る」 → 「一匹のオレンジ色の太った猫が夕日の差す草原で楽しげに駆ける」
- 遅延: 数百 ms
- 扩写後は **必ずもう一度安全審査を通す**。扩写が新たなリスクを持ち込まないようにするため

## 十六、Cost Reduction（降本戦略）

| 戦略 | メカニズム |
| --- | --- |
| **FP8 量子化** | 显存占有の削減 + スループット向上 |
| **offline GPU の呼び戻し** | ピーク時に訓練タスクを一時停止し、GPU を推論へ回す |
| **時差調度** | 北米のビジー時 → 欧州深夜の空きマシンルームへ調度 |

| **長タスクを抢占しない** | 60 GB 活性化の保存コスト > やり直しコスト |

| --- | --- | --- |

## 二十、核心結論と実戦アドバイス

### 核心結論
1. **GPU が絶対的な核心制約**: あらゆるアーキテクチャ上の決定は「いかに GPU を節約し/GPU 利用率を最大化するか」を軸に据える
2. **推論の途中で止めるな**: 60 GB の活性化を保存する IO 開銷は、やり直すより高くつく
3. **失敗ドメインを隔離せよ**: 多段の準入 + 前置安全で、GPU を無駄に消費するのを避ける
4. **時差を存分に活用せよ**: 全球のデータセンターは天然の「コスト arbitrage」だ

### 最適化の方向
- **Self-Attention の平方計算量**が最大の瓶頸
- **系列並列（CP4 + TP2）+ FP8 混合精度** が主力武器
- **Pipeline の異構ハードウェア**（CPU / 小 GPU / H100）がコスト差を広げる

### 設計時に先に計算しておくべきいくつかの勘定
- Capacity 見積もり（5M/日 → 50 平均 QPS → 200 ピーク QPS）、まず量級を掴む
- FLOPs 見積もり（2 × 参数 × seq_len × steps の公式を使う）、単回推論コストを判断する
- 中間活性化値の見積もり（60 GB → 「抢占しない」決定を導く）
- 「offline GPU の呼び戻し」は突発トラフィックへの一般的な対処法

## 二十一、システム設計への示唆

- **高価なリソース → 多段の関門**: GPU のような希少リソースの手前には三重の準入制御を積む
- **大きな状態は抢占しづらい**: worker の中間状態が GB 級を超えると、抢占コストが走り切らせるより高くなりうる
- **非同期 + ステートマシン**: 長時間タスクは一律で非同期化し、ステートマシンを明示的にモデリングする（Ready → Running → Complete/Fail/Retry）
- **異構 Pipeline で段を分ける**: 各ステージ（前処理/推論/後処理）に異なるハードウェアを当て、高価な GPU に安い仕事をさせない
- **跨地域の時差調度**: GPU 系業務に固有のコスト武器
- **Capacity Estimation が設計の第一歩**: 経験上、必ず先に QPS、FLOPs、ストレージを計算する。量級が明確になって初めて後続の決定に根拠が生まれる
- **安全審査は「前置 + 後置」の二層が必須**: 単層では不十分。前置で入力を防ぎ、後置で生成を防ぐ
- **Self-Attention O(n²)**: 長系列タスクの根本的な瓶頸であり、系列並列が存在する理由
