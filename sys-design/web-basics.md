# Webサイトの基礎

> このノートは私がSystem Designを整理するうえでの出発点にあたる部分だ。重点は複雑なアーキテクチャを積み上げることではなく、まず最も基礎的なWebサイトのリクエストがどう流れるか、バックエンドシステムに最低限どんなコンポーネントが必要か、そしてその後のあらゆる複雑な設計がなぜこれらの基礎の上に成り立つのかを、整理して理解することだ。

## なぜここから整理するのか
経験上、System Designに触れ始めた頃に最も逸脱しやすいのは次の点だ。
- いきなり分散システムを語り出す
- 口を開けばキャッシュ、メッセージキュー、マイクロサービス
- だが最も基本的なWebサイトのリクエストが実際どう流れるのかを説明できない

だからこの種の「Webサイトの基礎」の役割は、簡単だからではなく、**土台を作る**ことにある。基礎のリンクが不明瞭なら、後で拡張・高可用・非同期化・シャーディングを語っても空理空論になりやすい。

## Webサイトの最も基本的なワークフロー
ユーザー視点で見れば、1回のWebサイトのリクエストは往々にしてとてもシンプルだ。
1. ユーザーがブラウザにURLを入力するか、ボタンをクリックする
2. ブラウザがHTTPリクエストを発行する
3. サーバーがリクエストを受け取り、ビジネスロジックを実行する
4. サーバーがHTML・JSON・その他のレスポンスを返す
5. ブラウザが結果をレンダリングする

従来型のWebサイトなら、サーバーは完全なHTMLを直接返すかもしれない。フロントエンドとバックエンドが分離したアーキテクチャなら、サーバーは通常JSONを返し、フロントエンドのページが自分でレンダリングする。

## 最小限動くWebサイトシステムには何が含まれるか
最小限動くWebアプリには、通常少なくとも次の部分が必要だ。
- **フロントエンドのページ / クライアント**：ユーザー操作の入口
- **Webサービス / APIサービス**：リクエストとビジネスロジックを処理する
- **データベース**：ユーザー・コンテンツ・関係などのデータを保存する
- **認証とセッションのシステム**：ユーザーの身元を識別する
- **ログと監視**：システムが正常かどうかを知る

これらのコンポーネントは基礎的だが、その後のほぼすべてのシステム設計問題は、最終的にこれらの拡張と組み合わせに戻ってくる。

## ブラウザ・サーバー・データベースはそれぞれ何を担うか
### ブラウザ / クライアント
主に次を担う。
- 画面の表示
- ユーザー入力の収集
- リクエストの発行
- 結果のレンダリング

### サーバー
主に次を担う。
- リクエストの受信
- パラメータの検証
- ビジネスロジックの処理
- データベースや他サービスの呼び出し
- レスポンスの返却

### データベース
主に次を担う。
- 永続データの保存
- クエリと更新のサポート
- ある程度の一貫性と信頼性の担保

経験上よくある問題は、この3層がそれぞれ何をすべきかを本当には分かっておらず、結果として責務をひとまとめにしてしまうことだ。

## 典型的な例：ソーシャル系プロダクトの投稿
「ユーザーが1件投稿する」を例にとると、リクエストのリンクはこうなりうる。
1. フロントエンドのページが投稿リクエストを送信する
2. APIサービスがユーザーの身元と入力内容を検証する
3. サーバーが投稿をデータベースに書き込む
4. サーバーが成功結果を返す
5. フロントエンドがページを更新し、新しい投稿を表示する

さらに拡張すれば、ここに次が加わるかもしれない。
- 画像アップロード
- コンテンツ審査
- メッセージ通知
- タイムラインへの配信
- 検索インデックスの更新

つまり、一見シンプルな「投稿」という動作も、その背後では多くのシステム設計問題を引き出す。だが最初の一歩は常に、最も基本的な書き込み経路を明確に説明することだ。

## なぜSystem DesignはWebサイトの基礎から始めるのか
なぜなら後に出てくる多くの古典的設計は、本質的にこの最も基礎的なモデルの上に複雑さを足しているだけだからだ。

例えば、
- ユーザーが増えれば、拡張を考える
- アクセスが遅くなれば、キャッシュとレイテンシ最適化を考える
- 読み書きの衝突が増えれば、データベース設計を考える
- 呼び出しチェーンが長くなれば、サービス分割と非同期化を考える
- 容量が大きくなれば、パーティションとレプリカを考える

言い換えれば、**複雑なシステムは何もないところから現れるのではなく、シンプルなWebサイトから一歩ずつ進化してくる**のだ。

## 実践でよくある誤解
### 1. いきなり複雑なアーキテクチャ図を描く
最も基本的なリクエスト経路すら整理できていなければ、複雑な図をいくら描いても説得力はない。

### 2. 技術用語を積み上げるだけ
例えばいきなりRedis、Kafka、ElasticSearch、Kubernetesを持ち出すが、それらがなぜこのシステムに出てくるのかを考え抜いていない。

### 3. 責務の境界が不明瞭
フロントエンド・サーバー・データベースがそれぞれ何を担うかを切り分けられていなければ、設計はとても散漫に見える。

### 4. 「ユーザーの動作」からシステムコンポーネントを導かない
経験上、System Designの最も自然な推論の進め方は、ユーザーのリクエストから出発し、一歩ずつバックエンド構造へと辿るものであって、逆向きにテンプレートを当てはめることではない。

## このノートの使い方
このノートを、その後のあらゆるシステム設計のベースラインとして使う。複雑なシステムに出会ったら、まず基礎の問いにすばやく立ち戻ればいい。
- ユーザーの動作は何か
- リクエストのリンクはどう流れるか
- データはどこに置かれるか
- どの層が最初にボトルネックになるか

こうすれば推論がふらつかない。

## まとめ
Webサイトの基礎は単なる「入門知識」ではなく、System Designの共通の地盤だ。本当に重要なのは複雑なアーキテクチャを持ってこられるかどうかではなく、まず最も基本的なWebサイトのリクエストを整理できるかどうかだ。すなわち、ブラウザがどうリクエストを発行し、サーバーがどう処理し、データベースがどうデータを保存し、そしてこれらの基礎コンポーネントが後にどのように一歩ずつ複雑なシステムへ進化していったのか、ということだ。
