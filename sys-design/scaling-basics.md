# システムのスケーリング

> このノートではシステムのスケーラビリティの基本的な考え方を扱う。重点は、トラフィック・データ量・ユーザー規模が増え続けるときにシステムが一般的にどの軸で拡張していくか、そしてデータ分割・機能分割・レプリカ追加という3つの手法がそれぞれどんな問題を解決するのか、という点だ。

## こういうシステムで解決したいこと
「システムをどう拡張するか」というのは、本質的には次の問いに答えることだ。
- 1台のマシンで捌けなくなったらどうするか
- 1つのDB・1つのテーブルに収まらなくなったらどうするか
- トラフィックが急増したらどこから手をつけるか
- 容量を増やしながら、可用性と保守性をどう保つか

つまりスケーラビリティの問題は単一の技術的な問題ではなく、**容量・アーキテクチャ・組織の在り方が一緒に進化していく**問題だ。

## システム拡張の代表的な3方向
経験上、まずは3つの考え方に収束させておくとよい。
1. **データ分割**
2. **機能分割**
3. **レプリカ追加**

この3つは排他的なものではなく、実際のシステムでは大抵同時に使われる。

## 1. データ分割（Data Partitioning / Sharding）
データ分割の発想は、どんどん大きくなる1つのデータセットを分けて、異なるマシンに異なるサブセットを担当させる、というものだ。

### よくある分け方
- ユーザーIDでシャーディング
- 投稿IDでシャーディング
- 注文IDでシャーディング
- 地理リージョンでシャーディング

### 何を解決するか
データ分割がもっとも直接的に解決するのは次の点だ。
- 1つのDBの容量が足りない
- 1台のマシンのストレージが足りない
- 単一ノードの読み書き負荷が高すぎる

### メリット
- ストレージ容量の上限を大きく引き上げられる
- 1テーブル・1DBあたりの処理能力を高められる
- データ規模が増え続けるシステムに向いている

### デメリット
- シャード横断のクエリが複雑になる
- シャードキーの選び方を誤るとホットスポットになりやすい
- 拡張後にデータ移行やリバランスが必要になることがある
- グローバルな一意性やトランザクション機能が難しくなる場合がある

### 実践で気をつけるべき点
データ分割を語るとき、「シャーディング（分割）」で止まってはいけない。次の点まで考え抜く必要がある。
- なぜこのキーで分けるのか
- ホットスポットは生じないか
- クエリパターンはシャードをまたぐか
- 拡張後にどうrebalanceするか

## 2. 機能分割（Functional Partitioning）
機能分割の発想は、業務モジュール単位でシステムを分け、異なるサービスに異なる責務を担当させる、というものだ。

例えばソーシャル系プロダクトなら、次のように分けられる。
- ユーザーサービス
- 投稿サービス
- タイムラインサービス
- 検索サービス
- 通知サービス

### 何を解決するか
機能分割が主に解決するのは次の点だ。
- モノリスがどんどん大きくなり、変更が難しくなる
- 業務ごとに成長速度が違う
- モジュールごとに性能やストレージへの要求が異なる
- チームの協業境界が曖昧になる

### メリット
- 各モジュールを独立してスケールできる
- 技術スタックやストレージを業務ごとにカスタマイズできる
- チームの境界がはっきりする

### デメリット
- サービス間の呼び出しチェーンが長くなる
- 分散システム特有の問題が出始める
- 運用コストが上がる
- 障害の切り分けが複雑になる

### どう判断するか
ボトルネックが「機能がどんどん増え、チームがどんどん大きくなり、モノリスのコードがどんどん保守しづらくなる」ことなら、いきなりデータ分割をやるより機能分割の方が自然なことが多い。

## 3. レプリカ追加（Replication / Replicas）
レプリカ追加の発想は、同じ種類のサービスやデータに複数のレプリカインスタンスを用意し、負荷を分担させる、というものだ。

典型的なやり方は次のとおり。
- Webサービスのマルチインスタンス展開
- データベースのリードレプリカ
- Redisのレプリカ
- CDN / エッジノードの拡張

### 何を解決するか
レプリカ追加が主に解決するのは次の点だ。
- 読み込みトラフィックが多すぎる
- サービスの可用性が足りない
- 単一インスタンスがボトルネックになりやすい

### メリット
- 実装の道筋が比較的シンプルなことが多い
- 読み多書き少のシナリオで特に効く
- ついでに可用性も上げられる

### デメリット
- レプリケーション遅延が出ることがある
- 一貫性の問題が顕著になる
- 書き込みトラフィックはレプリカでは解決しきれないことがある

### よくある判断
システムが「読み多書き少」なら、まずレプリカとキャッシュを思い浮かべるのは大抵理にかなっている。「データ量が大きすぎる」なら、分割を思い浮かべる方が自然だ。

## この3つの拡張手法はそれぞれ何を解決しているか
ざっくりこう理解できる。
- **データ分割**：「収まりきらない」「単一点が重すぎる」を解決する
- **機能分割**：「システムが雑多すぎる」「責務が多すぎる」を解決する
- **レプリカ追加**：「アクセスが多すぎる」「可用性が足りない」を解決する

これは拡張の考え方の枠組みをすばやく作るのにとても向いた分類だ。

## スケーラビリティのよくある進化順序
多くのシステムは最初から複雑なアーキテクチャを採るわけではない。より一般的な進化の道筋はこうだ。
1. まず単一マシン
2. 次にマルチインスタンス + ロードバランサ
3. 次にキャッシュとDBレプリカを追加
4. 次にデータまたは機能で分割
5. 最後により複雑な分散段階へ進む

この背後にある経験則は、**アーキテクチャの複雑さは本当のボトルネックに合わせて増やすべきで、先回りして設計するものではない**、ということだ。

## 設計時にどの層から拡張するかをどう判断するか
経験上、まずいくつか問いを立てる。
- ボトルネックは計算・ストレージ・ネットワークのどれにあるか
- 読み込み負荷が大きいのか、書き込み負荷が大きいのか
- データ量が大きすぎるのか、機能が複雑すぎるのか
- すでにホットユーザー / ホットkey / ホットパーティションが出ていないか

ボトルネックをまず特定してこそ、その後の拡張手段に意味が出てくる。

## よくあるトレードオフ
### 広げれば広げるほど複雑さは増す
システムの容量は上がるが、その代わりに、
- 一貫性が難しくなる
- 障害面が広がる
- 呼び出しチェーンが長くなる
- 観測やデバッグのコストが高くなる

### すべてのシステムが最初から分散である必要はない
ボトルネックがまだ出ていないなら、いきなりシステムを十数個のサービスに分割する必要はない。

## まとめ
システムを拡張するとき、もっとも代表的な3つの手法はデータ分割・機能分割・レプリカ追加だ。これらはそれぞれ異なる種類のボトルネック、つまりデータが大きすぎる・責務が多すぎる・読み込みトラフィックが高すぎる、に対応している。本当に成熟したやり方は、これらの用語を機械的に当てはめることではなく、システムの現状のボトルネックに応じてどの層から拡張するかを判断でき、かつ各拡張手法がどんな新しい複雑さを持ち込むかを理解していることだ。
