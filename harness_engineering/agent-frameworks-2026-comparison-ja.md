# Agent フレームワーク 2026 競争地図 — 5 大アーキテクチャパラダイムの徹底比較

> 競争構造を再形成しつつある 3 つの大きな潮流と、それを踏まえた 10 大フレームワークの選定指針。

---

## 第一部 — 競争構造を作り変える 3 つの出来事

### 1. MCP の標準化

**MCP（Model Context Protocol）** は OpenAI が 2024 年末に提唱した、**大規模モデルと外部ツール / データソースの相互作用を定義する標準プロトコル**。

> Agent のツール呼び出し領域における **USB インターフェース** だと考えればよい。

MCP 登場以前は、各フレームワークがそれぞれ独自のツールインターフェース定義を持っていた。LangChain で書いたツールは CrewAI に持っていくと書き直しが必要だった。

しかし **2026 年 3 月** 時点で、主要 10 フレームワーク中、**少なくとも 8 つが MCP をネイティブサポート** している。

これは何を意味するか:

> **ツールエコシステムの参入障壁が消えつつある。**

MCP 標準でラップされたツールは、ほぼあらゆるフレームワークでそのまま利用できる。したがって、フレームワーク選定における「どれだけのツールに対応しているか」という観点の重みは **大きく低下** している。ツールレイヤは標準化によって平準化されたためだ。

### 2. A2A プロトコルの興隆

**A2A（Agent-to-Agent Protocol）** は Google が 2025 年に提唱した、**異なるフレームワークで構築された Agent 間の通信を定義するプロトコル**。

- MCP が解決するのは: **Agent がツールをどう使うか**
- A2A が解決するのは: **Agent が Agent とどう話すか**

現在 A2A をネイティブサポートしているのは:

- Microsoft Agent Framework
- Google ADK
- AgentScope

CrewAI と LangGraph はコミュニティプラグイン経由で部分的にサポート。

A2A の普及スピードは MCP に及ばない。なぜなら、その真の価値は **組織横断の Agent 相互運用** にあるためだ。たとえば自社の Agent がパートナー企業の Agent と協調するようなケース — このような変化はより長い導入サイクルを要する。

しかし方向性は明確だ:

> **Agent エコシステムは、フレームワークの孤島から相互運用ネットワークへと移行している。**

### 3. Engineering 理念のアップグレード

2025 年に新たな共通認識が広まった:

> **Prompt Engineering だけでは不十分。Context Engineering もやる必要がある。**

つまり、**大規模モデルのコンテキストウィンドウへ送り込む情報の品質と構造を、丹念に管理する** ことだ。

この変化が以下の方向のフレームワーク革新を直接推進した:

- **メモリ圧縮**
- **コンテキストフィルタリング**
- **動的ツール選択**
- **情報の優先度管理**

## 第二部 — フレームワーク差別化の本質的論点

この 3 つの出来事はあわせて、**「良い Agent フレームワークとはどんな姿か」** を再定義している。

- ツール呼び出しは **MCP** で標準化
- Agent 間通信は **A2A** で標準化

では、フレームワーク間に残された真の差別化競争は何か？

> **答え: アーキテクチャパラダイムとオーケストレーション能力。**

## 第三部 — 5 大アーキテクチャパラダイムの正面対決

### パラダイム 1: グラフステートマシン

**代表選手: LangGraph**

LangGraph の哲学を 6 文字に要約すると:

> **抽象を減らし、コントロールを増やす。**

提供されるコア原語は実にシンプル:

- **Node**
- **Edge**
- **State**

開発者はこの 3 つのブロックで、任意の複雑な実行グラフを自ら組み立てる。設計のインスピレーションは **Google Pregel** と **Apache Beam** に由来する。

LangGraph はアーキテクチャ判断を **一切代行しない**。Agent がどう思考しどう協調するかも事前に決め付けない。開発者は以下を自分で設計する:

- 状態スキーマ
- オーケストレーションロジック
- エラーハンドリング
- 中断と復帰

LangChain チーム自身の言葉:

> 「使いやすさ」と「本番投入準備」の二択を迫られたとき、私たちは後者を選んだ。

これが意味するのは:

> **柔軟性は最高、学習コストも最高。**

実際のフィードバックによれば、経験のあるチームでも **2–3 週間** はかかる、本番グレードの LangGraph コードを書けるまでに。

ただしリターンも最大級:

- **Checkpointer** 内蔵で状態永続化対応
- **「タイムトラベル」**: 任意の過去状態に戻ってデバッグ可能
- **interrupt** 内蔵: 任意のノードで実行を一時停止し、人手承認を待てる
- 復帰時はブレークポイントから継続
- **トークンレベルのストリーミング**、追加オーバーヘッドはほぼゼロ

ユースケース例:

- **Klarna** が CS Agent 構築に使用
- **Replit** がコード生成ワークフロー編成に使用
- 多くのデータエンジニアリング AI アシスタントが採用

#### コア能力スコア

| 観点 | スコア |
|---|---|
| オーケストレーション能力 | 5 |
| 本番デプロイ成熟度 | 5 |
| エコシステムの豊富さ | 5 |
| 可観測性 | 5 |
| 柔軟性・カスタマイズ性 | 5 |
| 習得難易度 | 2 |

LangGraph は Agent フレームワーク界の **Linux** にたとえられる:

> 強力、しかし本当に理解している人を要求する。

### パラダイム 2: ロール駆動

**代表選手: CrewAI**

CrewAI の哲学はより直感的:

> **実際のチームを作るように、Agent チームを作る。**

各 Agent には以下が与えられる:

- **Role**
- **Goal**
- **Backstory**

自然言語に近い形で、個性と専門性を持った AI ロールを定義できる。

> **役を定義 → タスクを割り当て → チームを起動。3 ステップで完了。**

CrewAI は **LangChain から完全に独立** して構築されており、外部依存は無い。

2 つのオーケストレーション方式を提供:

- **Crew**: 自律協調を重視、Agent は仲間にタスクを委譲できる
- **Flow**: イベント駆動で精密な制御パスを提供

現状の数字:

- GitHub Stars: **45,900+**
- **10 万人以上** の開発者が公式コースで認証取得済み

習得難易度スコアは **ほぼ満点の「やさしい」**。

ただし、使いやすさには代償がある:

- コアフレームワークがまだ **0.x** バージョン
- API 変更が頻繁
- 低レベルの制御力は LangGraph に及ばない
- 複雑な条件分岐や状態管理の表現力にも限界

**適するシナリオ:**

- コンテンツ制作パイプライン
- 市場調査
- 高速プロトタイピング
- 多役 Agent 協調デモ

**適さないシナリオ:**

- 極限の状態制御が必要な金融取引システム
- 正確なエラー復帰が必要な長時間実行ワークフロー

---

### パラダイム 3: イベント駆動 / データ中心型

**代表選手: LlamaIndex、AgentScope**

#### 3-1. LlamaIndex

LlamaIndex のコア哲学:

> **データは Agent の知能の出発点である。**

他のフレームワークが「オーケストレーション」「協調」を競う一方、LlamaIndex は別の層を深掘りしている:

> **正しいデータを、正しいタイミングで、正しいフォーマットで、大規模モデルに送り込む方法。**

データレイヤが非常に強力:

- **300+ コネクタ**
- **LlamaParse** ドキュメント解析エンジン
- **130+ ドキュメントフォーマット** 対応
- 複雑なネストテーブル、スキャン画像、手書きメモまで対応

オーケストレーション層は **Workflow 1.0** のイベント駆動路線を採用し、**ステップごとに組合せ可能** に実行する。

「データ集約型 Agent」という垂直領域では **濠が極めて深く、ほぼ代替不可能**。

典型シナリオ:

- プライベートエクイティが複雑な金融文書を処理
- 保険会社が証券分析を実施
- 製造業が技術仕様書から洞察を自動抽出

#### 3-2. AgentScope

AgentScope は **アリババ通義実験室** 発。コア理念:

> **透明、コントローラブル。**

多くのフレームワークが複雑さを隠そうとするのに対し、AgentScope は逆に強調する:

- 全 Prompt が可視
- 全 API 呼び出しが可視
- 全意思決定ステップが可視

特徴的な強み:

- **モデルファインチューニング内蔵**
- フレームワーク内で **Agent 強化学習ファインチューニング** 可能
- **Python と Java の両バージョン** を提供
- 国内エコシステムへのネイティブ対応: 通義千問、アリババクラウド Function Compute、DingTalk 等

中国国内でのエンタープライズ私有化、高コンプライアンス要求、国産モデル路線の場合、AgentScope は自然な候補となる。

---

### パラダイム 4: SDK 軽量フレームワーク

**代表選手: OpenAI Agents SDK、PydanticAI**

#### 4-1. OpenAI Agents SDK

フレームワーク全体がほぼ **3 つのコア概念** のみ:

- **Agent**
- **Handoff**
- **Guardrails**

**Handoff** が最も重要な設計上のハイライト。Agent がタスクを完了した後、より適した別の Agent に **制御を引き継ぐ** ことができる — CS 電話の「転送」のように。

導入の敷居は極めて低く、**5 行のコード** で Agent を動かせる可能性がある。

OpenAI 公式提供のため、OpenAI 自社モデル、Responses API、音声リンク、ツールエコシステムとの統合が非常に自然。

ユニークなハイライト:

> **Real-time Voice Agents をネイティブサポート。**

具体的には:

- 割り込み検出
- リアルタイムストリーミング音声
- 音声インタラクション体験の最適化

欠点も明確:

- 永続実行能力に欠ける
- 成熟した checkpoint メカニズム無し
- オーケストレーションパターンが限定的
- 複雑なループや並列が苦手

**適するシナリオ:**

- 高速立ち上げ
- 軽量な多 Agent 協調
- 音声 Agent
- OpenAI エコシステムと深くバインドするシナリオ

#### 4-2. PydanticAI

PydanticAI は別の路線:

> **型安全。**

スローガンを言い換えるなら:

> **FastAPI の体験を Agent 開発に持ち込む。**

最大の価値:

- すべての入出力を **Pydantic モデルで検証**
- 構造化出力の自動パース
- 失敗時の **自動リトライ**
- **25+ のモデルプロバイダ** をサポート
- モデル切替時にビジネスロジックを書き直さなくて済む

**mock とテストツール** も内蔵し、**決定論的なテスト** をサポート — これは CI/CD パイプラインでは救命級。

> PydanticAI は **「型安全 Agent 基盤層のデファクトスタンダード」** になる可能性が高いと評価する声が多い。

### パラダイム 5: ローコードプラットフォーム

**代表選手: Dify**

Dify は特殊な存在。「純粋なコードフレームワーク」ではなく、**完全な Web プラットフォーム** だ。

Python パッケージではなく、ブラウザ内で操作できる一式の能力を提供:

- ドラッグ&ドロップのワークフローエディタ
- モデル管理
- ナレッジベース管理
- API 公開
- アプリケーション監視
- デプロイ能力

つまり、**非エンジニアでも複雑な Agent ワークフローを構築できる**。

GitHub Stars は現在 **13 万超** で、これは多くの Agent フレームワークを圧倒する規模。

コア価値は技術的深さではなく:

> **AI アプリケーション開発の民主化。**

天井も明確:

- ビジュアルエディタ自体の表現力に縛られる
- 複雑なカスタマイズ要件は結局コードに戻る

## 第四部 — 重要な観察: ゼロサムではなく **ニッチ分化**

5 大パラダイムを並べると気づくのは、**それらが同じ次元で競っていない** ということ。

| フレームワーク | 勝っている領域 |
|---|---|
| LangGraph | コントロール力 |
| CrewAI | 使いやすさ |
| LlamaIndex | データ処理能力 |
| PydanticAI | 型安全 |
| Dify | 参入障壁の低さ |

> ゼロサムゲームではなく、**生態的ニッチの分化**。

## 第五部 — 大手の企業統一フレームワーク 2 強

### 1. Microsoft Agent Framework

これは **2025 年 10 月以降、最も重要な業界イベントの 1 つ**。

マイクロソフトは以下を **統合** した:

- **AutoGen**: 学術界先端の多 Agent 研究フレームワーク
- **Semantic Kernel**: エンタープライズ寄りでエンジニアリング・安定性重視

統合後の Microsoft Agent Framework は両方の遺伝子を継承:

- AutoGen の **グループチャット、ディベート、リフレクション** 等の先進的多 Agent パターン
- Semantic Kernel の **エンタープライズセキュリティ、ガバナンス、エンジニアリング能力**

特徴的な強み:

- **.NET と Python の両方を主流フレームワークでネイティブサポートする唯一の存在**
- **A2A、MCP** 等の主要標準をネイティブサポート
- **Azure AI** との深い統合

**2026 年 2 月時点で Release Candidate に到達**。**2026 年 Q1 末に正式リリース予定**。

> .NET 技術スタック企業にとっては、**ほぼ自然な、しばしば唯一の候補**。

### 2. Google ADK

Google ADK は **多言語路線**:

現状のカバー範囲:

- Python
- TypeScript
- Java
- Go バージョンも開発中

提供する Agent タイプ:

- **LLM Agent**: 知的推論
- **Sequential / Parallel / Loop Agent**: 決定論的オーケストレーション
- **Base Agent**: 低レベルカスタム拡張

**Vertex AI、Google Cloud とのネイティブ統合** が、Google Cloud 系企業の自然な選択となる鍵。

ただし制約も明確:

- バージョンはまだ **0.x**
- コミュニティ規模は LangGraph / CrewAI に大きく劣る
- Google エコシステム外では柔軟性が顕著に低下

## 第六部 — 10 フレームワークの選定: 4 ステップ意思決定木

すべての特性を覚える必要はない。**4 つの質問** に答えるだけでよい。

### Step 1: チームの技術スタックは?

| 技術スタック | 推奨 |
|---|---|
| **.NET / C#** | **Microsoft Agent Framework**。ほぼ他に選択肢無し |
| **Java** | **Google ADK** または **AgentScope Java 版** |
| **TypeScript** | **OpenAI Agents SDK の JS/TS 版**、または **LangGraph.js** |
| **多言語混在チーム** | **Google ADK** がカバレッジ最広 |
| **純 Python** | 選択肢が最多、Step 2 へ |

### Step 2: コアユースケースは?

| シナリオ | 推奨 |
|---|---|
| データ集約型 RAG / 文書 QA | **LlamaIndex**（実質敵なし） |
| ロール化された多 Agent 協調 | **CrewAI**（最も直感的） |
| 複雑な状態ワークフロー | **LangGraph**（ゴールドスタンダード） |
| 高速プロト / MVP | **CrewAI** または **OpenAI Agents SDK** |
| 企業向けローコード AI プラットフォーム | **Dify** |
| 中国国内企業の私有化デプロイ | **AgentScope** または **Dify** |

### Step 3: どのクラウドにデプロイ?

| クラウド | 推奨 |
|---|---|
| Azure | **Microsoft Agent Framework** |
| Google Cloud | **Google ADK** |
| アリババクラウド | **AgentScope** |
| 自前ホスト / 私有化 | **Dify** |

特定のクラウド志向が無ければ、シナリオとチーム技術スタックに戻る。

### Step 4: モデルへのこだわりは?

| 志向 | 推奨 |
|---|---|
| OpenAI エコシステムに強く依存 | **OpenAI Agents SDK** |
| Gemini 依存 | **Google ADK** |
| 国産モデル使用（通義千問、文心一言等） | **AgentScope / Dify** |
| モデル非依存 / 最大の柔軟性 | **PydanticAI** |

この 4 ステップで、選択肢はおおよそ **1–2 個** に絞られる。

## 第七部 — 見落とされがちな観点: 隠れたコスト

10 フレームワークすべてのコアコードは、ほぼオープンソース無料。

しかし:

> **「オープンソース無料」と「本番運用可能」の間には、巨大な隠れコストが横たわる。**

具体例:

- **LangGraph**: 本番投入時、多くのチームが付帯プラットフォーム能力に依存する
- **LlamaIndex**: 高品質なドキュメント解析能力はしばしば **有料**
- **Dify**: OSS 版と商用版で機能差がある
- **Microsoft Agent Framework / Google ADK**: いずれも各社クラウドの **有料サービスに深くバインド**

比較的、**PydanticAI** の隠れコストは低い。真に軽量な部類。

**フレームワーク選定時に「オープンソース無料」の 4 文字だけで判断してはいけない**。次を確認すべき:

> **本番投入後、主要能力を有料で補う必要があるかどうか。**

そうしないと、システムが完成してから「コア機能はお金を払わないと得られない」と気づき、プラットフォームに **ロックインされる**。

## 第九部 — トレンド判断

### 1. フレームワーク数はさらに減少する

2025 年のマイクロソフトによる AutoGen と Semantic Kernel の統合は、明確なシグナルを発している:

> **将来、フレームワーク数はさらに減り、生き残るものはより強くなる。**

### 2. コア能力はモジュール化される

LlamaIndex が Workflows を独立パッケージ化したのも、別のトレンドを反映している:

> **フレームワークのコア能力は、独立に切り出されて再利用されていく。**

将来、必ずしも「全部入りフレームワーク」を使う必要はなく、**そのうちの 1 レイヤだけ取り出して自分のシステムに組み込む** こともできる。

### 3. マルチモーダルが Agent の標準装備に

音声と視覚は Agent の基礎能力になりつつある:

- OpenAI Agents SDK の **Real-time Voice**
- Google ADK の **双方向ストリーミング音声 / 映像**
- より多くのフレームワークがマルチモーダル入力をサポート

> **マルチモーダルは付加機能ではなく、基本要件になる。**

## 第十部 — 中国国内エコシステム: ギャップと優位の併存

**ギャップ（客観的に存在）:**

- 国際コミュニティでの影響力は LangGraph / CrewAI に及ばない
- 英文ドキュメント品質、サードパーティ統合の幅で劣る

**優位（実在する）:**

- **Dify** の GitHub Stars は非常に高い
- **AgentScope** のモデルファインチューニング統合は独自性がある
- 国内フレームワークは **国産モデルへの対応がよりネイティブ**
