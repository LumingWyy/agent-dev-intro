# Prompt から Harness へ: なぜ Agent の本番運用で焦点が Harness Engineering に移っているのか

この 2 年、Agent 開発をめぐる議論の重心はかなり明確に移ってきました。最初は Prompt Engineering、その次に Context Engineering、そして 2026 年上半期に入ってからは、Harness Engineering に注目が集まっています。

Anthropic、OpenAI、Google DeepMind、Stripe、Windsurf などの公開事例を見ると、共通した認識が見えてきます。Agent を本番環境に載せられるかどうかを決めるのは、Prompt の巧拙だけではありません。使うモデルだけでもありません。むしろ重要なのは、モデルの外側にどのような実行システムを組んでいるかです。

まず `Harness` という言葉そのものを押さえておく必要があります。英語で harness は、本来は馬具やハーネス、つまり馬の力を人が制御可能な形で使うための装備を指します。この比喩を Agent に当てはめると、モデルは強力な「馬」であり、人間は進む方向を決める存在であり、Harness はその力を安全かつ有用な仕事へ導くための制御装置だと言えます。

ここでいう Harness とは、モデルのまわりに構築する「仕事の進め方の仕組み」だと捉えるとわかりやすいです。Harness Engineering は、モデルに代わって考えるものではなく、モデルがどう働くかを制約し、誘導し、検証するための仕組みを設計します。実際のエンジニアリングでは、タスク分解、コンテキスト管理、ツール編成、権限制御、状態の引き継ぎ、結果の検証、失敗時の復旧、人間へのエスカレーションといった要素が対象になります。

言い換えると、Prompt Engineering が扱うのは「どう指示するか」、Context Engineering が扱うのは「何を見せるか」、Harness Engineering が扱うのは「どのような実行機構の中で働かせ、どうやって完了を保証するか」です。

## なぜ 2026 年初頭に一気に注目されたのか

第一の理由は、モデル性能が一定水準を超えたことで、差分の中心がシステム設計に移ったからです。モデル自体は十分に強くなりましたが、長時間タスク、複数ステップ、複数ツール連携といった現実の業務で露呈するボトルネックは、単発回答の品質ではなくシステムの安定性にあります。

高レベルな指示を 1 つ与えるだけでは、強いモデルでもよく似た失敗をします。すべてを一度に終わらせようとしてコンテキストを使い切る、次のセッションで未完了なのに完了したと誤認する、検証をせずに作業を打ち切る、といった挙動です。こうした問題は、より強いモデルに置き換えただけでは解消しません。

第二の理由は、多段タスクで成功率が急速に落ちることです。1 ステップごとの成功率が 95% でも、20 ステップ直列にすると端から端まで成功する確率は約 36% まで下がります。Agent が「普段はほとんど正しく動いているのに、実運用では最後まで完遂しない」と感じられる背景には、こうした累積誤差があります。

第三の理由は、モデル自体の差別化が縮小しつつあることです。主要モデルの性能差が相対的に小さくなるほど、優位性を生み出すのはモデル API そのものではなく、モデルの周辺に構築された運用システムになります。その意味で、Harness は新しい競争優位の源泉として見られ始めています。

## Prompt、Context、Harness の関係

この 3 つは競合概念ではなく、階層的に積み上がるものとして理解するのが自然です。

Prompt Engineering は、モデルへの指示表現を最適化するものです。つまり「どう聞くか」を扱います。Context Engineering は、入力をどう組み立て、どの情報を渡すかを扱います。つまり「何を見せるか」を最適化します。Harness Engineering はそこからさらに一段広く、モデルが使えるツール、守るべき制約、状態の維持方法、検証方法、失敗後のリカバリ、人間への制御返却までを扱います。

より端的に言えば、Harness は Context を含み、Context は Prompt を含みます。Prompt が局所的な指示であり、Context が作業メモリだとすれば、Harness はその全体を動かす実行システムです。

## 先行するチームは何をしているか

Anthropic の実践は象徴的です。初期の構成では、長時間タスクを 2 種類の役割に分けていました。初期化 Agent は環境構築、要件の展開、進捗ファイルの作成、Git ベースラインの準備を担当し、後続のコーディング Agent は各セッションで進捗を読み込み、既存テストを実行しながら機能を前に進めます。ここで重要なのは、記憶を会話の内部に頼らず、進捗ファイル、構造化された要件、Git 履歴といった外部成果物として保持した点です。

2026 年 3 月には、Anthropic はこれを Planner、Generator、Evaluator の 3 役構成に発展させました。そして明確に示されたのが、「生成者に自分の成果を評価させても安定しない」という知見です。独立した評価役を別に用意した方が、品質検証を工学的に設計しやすい。これは Harness 設計におけるかなり重要な示唆です。

OpenAI の実践は、より本番システム寄りです。重要なのは、Agent に「もっと頑張れ」と求めるのではなく、Agent が欠いている能力をシステムとして補う発想です。コードベース知識、実行時情報、可観測性データへのアクセスを増やし、アーキテクチャ制約はモデルの善意ではなく deterministic な仕組みで守らせ、さらにバックグラウンドで不整合やアーキテクチャ逸脱を掃除する仕組みを回す。これは静的な Prompt 集ではなく、継続的に改善される Harness そのものです。

Google DeepMind も別の言葉でほぼ同じ方向に進んでいます。研究 Agent や ADK の構成を見ると、Generator、Verifier、Revisor のように生成と検証を分離した構造が現れています。Anthropic の Planner、Generator、Evaluator と高い対応関係があり、これは偶然というより、Agent を本番運用に近づけようとしたときに自然に収束する設計パターンだと考えた方がよいでしょう。

そのほかの事例も実務上の示唆が多くあります。よくある誤解の 1 つは、Agent に多くのツールを与えるほど賢くなるという考えです。しかし実際には、ツールが多すぎると探索空間が広がりすぎて、冗長な呼び出しや無駄な経路選択が増えます。むしろツールを絞り、解空間を狭めた方が、成功率と効率が上がるケースが少なくありません。

Stripe のアプローチも参考になります。Agent に必要なのは、寄せ集めのツール群ではなく、人間のエンジニアに近い作業環境そのものだという考え方です。隔離されたサンドボックス、十分に準備された実行環境、明確な権限境界、そして業務コンテキストにアクセスできる内部ツール群。これらをまとめて与えることが、Harness の質を左右します。

## 成熟した Harness は何を含むのか

各社の実践をまとめると、成熟した Harness には少なくとも 6 つの層があります。コンテキスト工学と知識管理、ツール編成と権限設計、検証と制約の仕組み、状態管理と記憶の持続、可観測性とフィードバックループ、人間による介入とライフサイクル管理です。

この中で特に見落とされやすいのが検証レイヤーです。Harness と単なる Agent スキャフォールドの違いは、モデルとツールを接続したかどうかではありません。モデルの振る舞いを deterministic に制約し、成果物を検査し、高リスク操作を止め、失敗の情報を再びシステムへ戻せるかどうかにあります。これがなければ、Agent は「実験的に動くシステム」のままです。これがあって初めて、「本番に近い運用システム」に近づきます。

## これは単なる「古い酒を新しい瓶に入れた」ものなのか

一部はその通りですが、それだけではありません。Harness Engineering が再利用している実践は多くあります。テストハーネス、CI/CD、Lint、権限分離、可観測性、ジョブ編成などは、どれも既存のソフトウェア工学に根ざしたものです。その意味で、完全な新発明ではありません。

ただし、扱う対象は従来と大きく異なります。従来のソフトウェア工学が主に制御してきたのは deterministic なコード実行でした。一方で Harness Engineering が制約しようとしているのは、確率的に振る舞う推論システムです。そのため、検証、リトライ、復旧、境界制御の設計には、従来より複雑な考え方が必要になります。

さらに重要なのは、適切な制約が Agent の能力を落とすどころか、むしろ引き上げる場合があるという点です。ツールを減らす、行動範囲を限定する、生成と評価を分ける、厳密な検証を挟む。こうした「自由度を減らす」設計が、結果的に成功率と品質を高めるというのは、かなり特徴的な現象です。

また、コードベースそのものが Harness の一部になり始めている点も見逃せません。モジュール境界、命名規約、ディレクトリ構造、説明文書は、人間の可読性だけでなく、Agent の可理解性や可推論性にも影響します。今後は「人間に読みやすいコード」だけでなく、「Agent が推論しやすいコードベース設計」が重要になる可能性があります。

## 注意すべき点

この概念にはリスクもあります。まず、Harness という言葉の適用範囲が広がりすぎると、何でも Harness と呼べてしまい、実務上の精度が落ちます。次に、過度なエンジニアリングです。モデルが進化すれば、過去のモデルの欠点を補うために作った仕組みが、将来的には不要な複雑さになる可能性があります。

また、現在の成功事例の多くはツール提供者自身による発信であり、独立した再現実験や定量的な比較はまだ十分ではありません。さらに、Harness は性能を増幅するだけでなく、リスクも増幅し得ます。マルチ Agent、複雑なツール連携、高度な自動化は、同時に異常の伝播範囲を広げる可能性があります。

## チームへの示唆

Harness Engineering を流行語ではなく、本番運用のための設計視点として捉えるなら、実務上の示唆は明確です。

第一に、Agent の失敗を Prompt の問題だけに還元しないことです。多くの失敗は、表現ではなく実行機構の欠如から起きています。

第二に、最低限の deterministic な検証レイヤーを早めに整備することです。ルール、テスト、権限、レビュー、停止条件を、モデルの自律性の外側に用意する必要があります。

第三に、Harness は最初から交換可能で観測可能な構成として設計した方がよいということです。モデルは必ず変化します。良い Harness は複雑であることではなく、モデル進化に合わせて不要な補助輪を外し、本当に残るべき仕組みを維持できることに価値があります。

## おわりに

要点を一言でまとめるなら、Prompt はモデルにどう話しかけるかを決め、Context はモデルに何を見せるかを決め、Harness はモデルが現実の環境で安定して仕事をやり切れるかを決めます。

Agent 導入を進めるチームにとって、これから本当に難しくなるのは「モデルにできるようにすること」ではなく、「モデルが制御可能で、検証可能で、復旧可能な仕組みの中で、継続的に成果を出せるようにすること」だと言えます。
